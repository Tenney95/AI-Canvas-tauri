## 修复：点击"控制角度"按钮没反应

- **根因**：React Flow v12 在 `pointerdown` 阶段捕获节点拖拽事件。`ImageNodeToolbar` 只在 `onClick` 上做了 `stopPropagation`，但 `pointerdown` 事件先于 `click` 被 React Flow 截获，导致节点拖拽启动后 `click` 被取消
- **修复**：`ImageNodeToolbar.tsx` 新增 `blockPointer` handler（`stopPropagation` + `preventDefault`），工具栏容器 div 和按钮上加 `onPointerDown={blockPointer}`，防止 pointerdown 冒泡到 React Flow
- **二次修复**：`pointerdown` 上的 `preventDefault()` 在浏览器层面阻止了后续 `click` 事件的触发（根据 DOM 规范，pointerdown 的 preventDefault 会 suppress click），导致按钮 onClick 永远不执行 → 移除 `preventDefault()`，仅保留 `stopPropagation()`
- **三次修复（最终）**：React Flow v12 使用原生 DOM 事件监听器处理拖拽，React 合成事件的 `stopPropagation()` 无法阻止原生监听器。正确做法是使用 React Flow 内置的 `nodrag` 机制：给工具栏容器及所有子级容器添加 `nodrag` class → React Flow 内部识别该 class 后自动跳过拖拽、不消费事件 → click 正常触发。同时移除了之前添加的所有 `onPointerDown` 手动拦截代码
- **修复 import 路径**：`FreeAnglePanel.tsx` 中 `freeangle.css` 的引用路径少了一层 `../`（文件在 `src/components/nodes/shared/`，深三层），改为 `../../../styles/freeangle.css`
- **修复 3D 正方体拖拽反转 360°**：`handleCubePointerMove` 中 `normalizeAngle(startRot - dx * 0.5)` 会在跨 0/360 边界时跳变（如 359.5→4.5），CSS `rotateY` 看到 ~355° 瞬间变化造成视觉反转。修复：拖拽期间不对 rotation 归一化，让 rotation 自由累积（CSS 天然支持任意角度），仅在显示值和生成参数输出时做 `normalizeAngle`
- **修复垂直拖拽方向反转**：`handleCubePointerMove` 中 `dy` 系数从 `+` 改为 `-`，向下拖 → pitch 减小（向下俯），与直觉一致
- **实现 APIMart 角度控制 API**：新建 `src/services/apimartService.ts`，封装完整流程：① data URL 自动上传到 APIMart (`POST /v1/uploads/images`) → ② 提交图像生成任务 (`POST /v1/images/generations`, 带 rotation/pitch 映射的 camera prompt) → ③ 轮询 (`GET /v1/tasks/{task_id}`) → ④ 返回图片 URL 列表。`ImageNode.tsx` 的 `handleFreeAngleGenerate` 已对接：调用服务 → 下载结果图片转 data URL → 计算节点尺寸 → `store.addNode()` 创建新图片节点

## 架构：项目本地文件存储 — 修复拖入视频+图片 OOM

**问题**：Tauri 端同时拖入图片和视频时，所有文件被读入内存转 base64 data URL。视频可数百 MB，base64 膨胀 33%，导致 `Error code: Out of Memory` 崩溃。

**方案**：媒体文件不再读入内存 → 直接拷贝到 `{appLocalDataDir}/data/{projectId}/` 目录，用 `convertFileSrc()` 生成 `asset://localhost/` URL 来渲染；仅文本文件仍读入内存（体积小）。

**改动文件（7 个）**：

- **`src/types/index.ts`**：`BaseNodeData` 新增 `filePath?: string` — 本地文件路径（卸载后重建 asset URL）
- **`src/services/fileService.ts`**：
  - 导入 `mkdir`, `exists`, `copyFile`, `stat` from `@tauri-apps/plugin-fs`
  - `getProjectDataDir(projectId)` — 获取 `{appLocalDataDir}/data/{projectId}/`
  - `ensureProjectDataDir(projectId)` — 自动创建目录
  - `copyFileToProjectData(srcPath, projectId)` — 拷贝文件处理重名冲突（`_1`...），>100MB 跳过拷贝直接用原路径
  - `saveDataUrlToProjectData(dataUrl, projectId, fileName)` — AI 生成图片保存到项目目录
  - `getAssetUrlFromPath(filePath)` — 封装 `convertFileSrc`
  - `uploadSourceFileToProject(accept, projectId)` — 上传并保存到项目目录
  - `MAX_MEDIA_FILE_SIZE = 100MB`
- **`src/store/useAppStore.ts`**：
  - `generateProjectId()` — `crypto.randomUUID()` UUID 固定不重复
  - `createProject` — 用 UUID + 调用 `ensureProjectDataDir`
  - `switchProject` / `initFromDb` — 确保数据目录存在
  - `pasteExternalFromDataTransfer` 的 `file-path` 分支 — 优先 `copyFileToProjectData`
- **`src/hooks/useNodeCreation.ts`**：Tauri 拖放循环中媒体文件优先 `copyFileToProjectData` → asset URL 存入 `imageUrl`/`videoUrl`/`audioUrl` + `filePath`；失败退回 base64

## 三次修复：Tauri v2 fs:scope 只授 READ 不授 WRITE

- **根因**：Tauri v2 的 `fs:scope` 只提供读取（READ）作用域，写入（WRITE）需要单独给写权限加上作用域。`writeFile` 报 `fs.write_file not allowed` 是 scope 问题不是权限问题
- **修复（`src-tauri/capabilities/default.json`）**：
  1. 移除 `"fs:allow-read"` 和 `"fs:allow-write"`（Tauri v2 中无效标识符）
  2. 新增 `{ "identifier": "fs:write-files", "allow": [{ "path": "$APPDATA/**" }] }` — 显式授予写入 AppData 的能力
  3. 新增 `{ "identifier": "fs:read-files", "allow": [{ "path": "$APPDATA/**" }, ..., { "path": "**" }] }` — 显式授予读取所有路径的能力
  4. 保留 `"fs:default"`（基础 app 作用域读写）+ `"fs:scope"`（只影响 read）
- **对比 Tauri v1**：v1 中 `fs:scope` 同时控制读写；v2 将读写作用域拆分，write 权限必须带上自己的 scope
