## 修复：粘贴的视频文件不保存到项目数据目录

- **问题**：用户通过 Ctrl+V 或拖放粘贴视频文件时，视频没有保存到 `{appDataDir}/data/{projectId}/` 目录，而是以 base64 或原始文件路径引用
- **根因**：`copyFileToProjectData` 中 `MAX_MEDIA_FILE_SIZE = 100MB`，视频超过此大小即跳过拷贝，只返回原始路径的 `convertFileSrc`。同时 `kind: 'file'` 分支的 `saveBinaryToProjectData` 缺少 try-catch 和日志

- **修复**：
  1. **`fileService.ts`**：`MAX_MEDIA_FILE_SIZE` 从 100MB → 2GB，覆盖绝大多数视频文件
  2. **`useAppStore.ts` pasteExternalFromDataTransfer**：`kind: 'file'` 分支的保存块增加 try-catch + console.log/warn/error 日志，方便定位失败原因
- **影响范围**：`copyFileToProjectData` 调用处（`useNodeCreation.ts` Tauri 原生拖放、`useAppStore.ts` kind: 'file-path' 分支）一并受益

## 修复：文件保存/管理系统跨平台兼容（Win/macOS/Linux）

### P0 严重 — `file://` URI 解析在 macOS/Linux 上丢失前导 `/`
- **问题**：`useAppStore.ts` 中 4 处 `decodeURIComponent(src.replace(/^file:\/\/\//, ''))`，在 Windows 上 `file:///C:/path` → `C:/path` 正确，但在 macOS/Linux上 `file:///home/user/file` → `home/user/file`（缺少 `/`），导致剪贴板粘贴文件功能彻底失效
- **修复**：新增 `fileService.fileUriToPath(uri)`，使用 `new URL(uri).pathname` 正确解析，同时处理 Windows（`/C:/...` → 去首 `/`）和 Unix 路径

### P1 — 硬编码 `/` 拼接路径
- **问题**：`fileService.ts` 多处 `${dataDir}/${fileName}` 硬编码 `/`，依赖 `appDataDir()` 返回正斜杠的非文档行为
- **修复**：新增 `fileService.joinPath(...segments)`，统一处理 `\\`→`/` 并去除重复分隔符，替换所有硬编码拼接

### P2 — 文件名清理只用 Windows 字符集
- **问题**：`sanitize = fileName.replace(/[<>:"|?*]/g, '_')` — 这些仅在 Windows 非法，在 macOS/Linux 上会过度清理合法文件名
- **修复**：新增 `fileService.sanitizeFileName(name)`，根据 `navigator.platform` 检测平台 — Windows 用 `/[<>:"|?*]/g`，Unix 只用 `/[/]/g`

### $APPDATA 变量（无需修改）
- `tauri.conf.json` 中 `assetProtocol.scope: ["$APPDATA/**"]` — Tauri 2 内置变量，自动映射：Windows→`%APPDATA%`，macOS→`~/Library/Application Support`，Linux→`~/.local/share`
