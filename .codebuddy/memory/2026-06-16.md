### 目标

### 修改

**`src-tauri/Cargo.toml`**:
- 新增 `ndarray = "0.16"`（后移除，ort 内置支持 `(D, Vec<T>)` 无需直接依赖 ndarray）

**`src-tauri/src/onnxrt.rs`** 重大重写:
- `SESSION` 改为 `Mutex<Option<(PathBuf, Session)>>` — 追踪已加载模型路径，路径匹配则复用
- `image_to_nchw_tensor()`: 读取图像 → 分离 R/G/B 平面 → NCHW [1,3,H,W] f32 [0,1]
- `nchw_tensor_to_image()`: NCHW f32 [0,1] → 分离平面 → 交错 RGB u8 → `image::RgbImage` → 保存到文件
- `image_upscale` 命令完整推理管线:
  1. 验证输入文件 + 模型文件
  2. `image_to_nchw_tensor` 预处理
  3. 锁定 SESSION → 获取 I/O 名称 → `ort::inputs!` 构造输入 → `session.run` 推理
  4. `try_extract_tensor::<f32>` 提取输出 → 复制为自有 Vec
  5. 块作用域确保 guard/session/outputs 释放后才执行 `nchw_tensor_to_image`
  6. 返回 JSON `{"output_path","input_size","output_size"}`
- 删除了未使用的 `with_session_ref` 辅助函数

### 踩坑记录
1. `ort::inputs!` 宏直接返回 `Vec` 而非 `Result`，不需要 `.map_err()`
2. `session.inputs()` / `session.outputs()` 是方法不是字段，`.name()` 也是方法
3. `session.run()` 需要 `&mut self`，不是 `&self`
4. `ValueRef` 的 `Deref` 返回引用绑定自身生命周期，无法在 `or_else`/`if let` 中跨作用域
5. `SessionOutputs` 持有 session 借用 → 不能显式 `drop(guard)`  → 用块作用域 `{}` 自动释放
6. `Cow<str>::as_ref()` 因 winnow 的 AsRef 实现导致类型推断歧义 → 改用 `.to_string()`

- `cargo check` 通过，零错误零警告

- 输出 clamp [0,1] → RGB u8 → 保存 PNG

## ONNX 推理挂起修复 — 两阶段锁 + 超时 (13:22)

### 问题
执行超分后 `POST /image_upscale` IPC 请求永久挂起，无响应无报错

### 根因分析
`get_or_load_session` 持有全局 `SESSION: Mutex` 锁期间执行 DirectML 初始化（`commit_from_file`），而 DirectML 首次编译模型可能极慢（30-60s）或因 GPU 驱动/模型兼容性卡死。一旦卡死：
1. 锁永不被释放
2. 后续所有 `image_upscale` 调用全部阻塞在 `.lock()` 上
3. 前端显示为"挂起"状态

### 修改

**`src-tauri/src/onnxrt.rs`**:

1. **`get_or_load_session` — 两阶段锁模式**:
   - 阶段 1：短暂持锁检查 session 是否已加载 → 释放锁
   - 阶段 2：**锁外**创建 Session（DirectML 编译，可能慢/卡死）
   - 阶段 3：短暂持锁存储 Session（含 double-check 防并发重复加载）
   - 效果：即使 Session 创建卡死，锁不会泄漏，后续调用可正常失败

2. **`image_upscale` — 5 分钟超时**:
   - `tokio::task::spawn_blocking` → 改为先拿到 JoinHandle，外围 `tokio::time::timeout(300s)` 包裹
   - 超时后返回明确错误："超分推理超时（5 分钟），DirectML 可能不兼容当前 GPU 或模型"

### 结果
- `cargo check` 通过
- 不再出现永久挂起，最坏情况 5 分钟后报错返回
- 并发调用不会互相死锁

## v2 子进程隔离架构 — ONNX GPU Worker (16:11)

### 动机
DirectML 初始化在虚拟显示适配器（MuMu/向日葵/Oray/etc）上会驱动级死锁，进程不可杀死。生产环境面向不特定用户，虚拟适配器非常普遍，不能在主进程碰 DirectML。

### 架构决策

**核心原则：子进程隔离**
- 所有 ONNX 推理在独立 `--onnx-worker` 子进程中执行
- 主进程永不初始化 DirectML，永不加载 ONNX Session
- Worker 崩溃/死锁 → 主进程毫发无伤，自动标记 CPU 并重试

### 新增文件

| 文件 | 作用 |
|---|---|
| `src-tauri/src/onnx_config.rs` | GPU 配置缓存（ep/device_id/device_name/probed_at）→ `%LOCALAPPDATA%/com.aicanvas.app/onnx_gpu_config.json` |
| `src-tauri/src/onnx_gpu.rs` | DXGI 适配器枚举 + 虚拟适配器过滤 + DirectML 探针推理（16×16 极小图测试） |
| `src-tauri/src/onnx_worker.rs` | Worker 主循环（stdin/stdout JSON lines 协议）+ probe/upscale/quit 三种 handler |

### 修改的文件

| 文件 | 改动 |
|---|---|
| `src-tauri/Cargo.toml` | `ort` 加回 `directml` feature；新增 `windows = "0.58"` (DXGI) |

| `src-tauri/src/onnxrt.rs` | 删除 `SESSION`/`get_or_load_session`/`run_upscale_tiled`；`image_upscale` 重写为 Worker 调度（探测→缓存→推理→DirectML 失败自动 CPU 重试） |
| `src/services/onnxService.ts` | 新增 `getOnnxGpuStatus()` / `GpuStatus` 接口 |

### IPC 协议（stdin/stdout JSON lines）

```
主 → Worker: {"id":"req-1","type":"probe","model_path":"..."}
Worker → 主: {"type":"ready",...}
Worker → 主: {"id":"req-1","type":"ok","result":{"gpu_supported":true,"device_id":2,"device_name":"NVIDIA..."}}

主 → Worker: {"id":"req-2","type":"upscale","ep":"directml","device_id":2,...}
Worker → 主: {"id":"req-2","type":"progress","done":5,"total":10,"percent":50}
Worker → 主: {"id":"req-2","type":"ok","result":{"input_size":"...","output_size":"..."}}
```

### GPU 选择策略
1. DXGI 枚举所有适配器
2. 跳过软件适配器（`DXGI_ADAPTER_FLAG_SOFTWARE`）
3. 跳过虚拟适配器（描述含 Virtual/Remote/Mirror/Idd/Indirect/Parsec/MuMu/Oray）
4. 优先 NVIDIA/AMD/Radeon/Arc 独显，按显存大小排序
5. 用 16×16 极小图执行 1 次完整 DirectML 管线测试

### 容错策略
- 首次使用自动探测 → 缓存结果
- DirectML Worker 超时/崩溃 → 自动标记 `ep=cpu` → 重试
- 之后该机器永久走 CPU（可手动删配置文件重新探测）
