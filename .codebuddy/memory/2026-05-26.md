## 修复: Tauri 全局快捷键在窗口失焦时仍拦截（二次修复）
- **文件**: `src/hooks/useKeyboardShortcuts.ts`
- **根因**: `@tauri-apps/plugin-global-shortcut` 是系统级全局快捷键。即使回调里只做 `isFocused()` 检查并跳过执行，OS 已经把按键交给了 Tauri，其他应用根本收不到按键事件。`ifFocused` 只能阻止执行逻辑，不能阻止 OS 级拦截。
- **修复**: 窗口聚焦时注册快捷键，失焦时取消注册。
  - 定义 `GLB_SHORTCUTS` 数组，结构为 `{ key, action }`
  - 通过 `win.onFocusChanged()` 监听窗口焦点变化
  - 聚焦 → `registerAll()` 依次注册所有快捷键
  - 失焦 → `unregisterAll()` 取消所有注册（释放给其他应用）
  - 初始化时检查当前焦点状态再决定是否注册
  - 清理时 unlisten + unregisterAll
- **结果**: AI Canvas 失焦时，快捷键在 OS 层面完全释放，其他应用正常接收按键

## 修复: 回归标准 React Flow source→target 连接模型
- **文件**: `TextNode.tsx`, `ImageNode.tsx`, `VideoNode.tsx`, `AudioNode.tsx`, `Canvas.tsx`
- **根因**: 所有 Handle 都是 `type="source"` 时，React Flow 在目标节点找不到 `type="target"` Handle，handle 匹配逻辑退化到始终选择左侧 handle，导致无论从哪侧拖拽都连到左侧
- **双 Handle（source+target 重叠）方案也失败**: React Flow 的 handle 匹配在重叠时选择了 target handle，但视觉效果（连接线起点）仍指向 DOM 中先出现的 handle（通常是左侧）
- **最终方案**: 回归 React Flow 标准范式——左侧 `type="target"`（接收），右侧 `type="source"`（发出）。移除 `isValidConnection`，`ConnectionMode` 改回 `Strict`
- 连接方向: 从任意节点的右侧拖拽 → 到任意节点的左侧释放
- 所有 source Handle 添加 `id="right"` 标识，ImageNode Bottom handle 也改为 `type="source"`

## 修复: 真正双向连线 — 每侧 source+target 重叠
- **需求**: 拖到哪个点就连在哪个点（左↔左、左↔右、右↔左、右↔右都支持）
- **方案**: 每侧放两个 Handle 重叠在相同 Position——可见的 `type="source"`（拖出线）+ 不可见的 `type="target"`（接收线）
  - 左侧: `id="left"` source (可见) + `id="left-target"` target (不可见)
  - 右侧: `id="right"` source (可见) + `id="right-target"` target (不可见)

- **CSS**: `.handle-target-invisible` — `opacity: 0` 保留 pointer-events，使 React Flow 能按位置就近匹配 target handle
- **Canvas.tsx**: 添加 `isValidConnection` 阻止自连接（`source !== target`），保持 `ConnectionMode.Strict`
- **关键细节**: 前次双 Handle 方案失败的原因可能是没正确设置各 target handle 的 id，或 CSS 影响了 pointer-events；本次重新实现且已验证无新 lint

- **五修(18:21)**: 改用 `sourceHandleId` 驱动——不再推测方向，直接根据拖出的是 `left`/`right`/`bottom` 哪个 handle 来决定新节点位置和端口连接：`left` → 新节点放左侧 + 连到其 `right` target；`right` → 新节点放右侧 + 连到其 `left` target；`bottom` → 新节点放下方 + 连到其 `left` target。间距 `gap = 50`。
- **六修(19:05)**: target handle ID 写错了——每个节点的 target handle ID 是 `left-target`/`right-target`/`bottom-target`（带 `-target` 后缀），不是 `left`/`right`/`bottom`。之前 edge 的 `targetHandle` 写了 `'left'`/`'right'`，React Flow 找不到对应 ID 的 target handle 就回退到自动放置，导致连线点错位。改为 `'left-target'`/`'right-target'`。
- **七修(19:09)**: 弃用 `sourceHandleId` 作为主方向信号——`connectionState.fromHandle?.id` 不可靠（每个节点同一位置有 source + target 重叠 Handle，connectionRadius=30，连接无效时 fromHandle 可能不准确）。改为**几何驱动**：用鼠标释放位置 `flowPos` 对比源节点边界 `srcX`/`srcRight`/`srcBottom`（`+10px` 阈值），清晰分为 `releasedRight`/`releasedLeft`/`releasedBelow` 三路。释放点在源节点内部时退到 `sourceHandleId` fallback，fallback 也失败时默认 right。`sourceHandleId` 仅在兜底逻辑中出现。
- **八修(19:16)**: 左侧拖出无连线——去掉 `addNode()` + `requestAnimationFrame(() => useAppStore.setState(...))` 两步走时序，改用 store 的 `addNodeWithEdge(node, edge)` 一次原子更新。同时优化左右放置时 `nodeY` 对齐源节点垂直居中 (`srcY + (srcHeight - newHeight) / 2`)，下方放置时 `nodeX` 对齐源节点水平居中 (`srcX + (srcWidth - newWidth) / 2`)，不再跟随鼠标 Y/X，让连线更整洁。
