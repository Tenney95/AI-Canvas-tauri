## 右键菜单子菜单 hover 修复
- **问题**：鼠标从父菜单行移到子菜单面板时，子菜单立即消失
- **根因**：
  1. Level 1 → Level 2：`onMouseLeave` 立刻清除 120ms 定时器，子菜单还没渲染就被掐掉
  2. Level 2 → Level 3：`onMouseLeave={() => openSubmenu(null)}` 立即将 hoverMenu 设为 null，所有子菜单消失
- **修复**：改为 `showSubmenu()`（立即显示 + 取消隐藏定时器）和 `hideSubmenu(backTo)`（250ms 延迟后回退到 `backTo`）
  - 所有 `onMouseEnter` → `showSubmenu(menu)`（立即展开，取消任何待执行的隐藏）
  - 所有 `onMouseLeave` → `hideSubmenu(backTo)`（250ms 后回退，给光标跨间隙留足时间）
  - 进入子菜单面板时 `showSubmenu` 会取消父行的 hideSubmenu 定时器
- **再次修复**：Level 2 子菜单的显示条件从 `hoverMenu === 'addNode'` 改为 `hoverMenu === 'addNode' || 'genNode' || 'srcNode'`，解决 hover 到 Level 3 时 `hoverMenu` 变为 `genNode/srcNode` ≠ `addNode` 导致 Level 2 消失
- Level 3 条件也同步简化：去掉冗余的 `|| 'addNode'` 保护（Level 2 现已始终可见）

## 图片上传后尺寸修正
- **问题**：上传图片后节点尺寸不按比例变化
- **根因**：高度计算公式 `nodeWidth - 24` 和 `+ 36` 错误
  - padding 补偿应是 `nodeWidth - 4`（`.image-node .node-preview.compact` 的 padding 为 2px 两侧）
  - `+ 36` 不该加（NodeLabel 是绝对定位，不在 body div 内）
- **修正**：`contentWidth = nodeWidth - 4; previewHeight = round(contentWidth / naturalRatio); newHeight = max(120, previewHeight + 4)`
- **效果**：16:9 横图约 159px（近默认 158px，变化极微），方形约 280px，9:16 竖图约 495px

## 右键菜单点击无效修复
- **问题**：右键菜单中点击生成节点/源节点不创建节点
- **根因**：`closeCtxMenu` 用 `mousedown` 事件监听，但只检查 `ctxMenuRef.current.contains(target)`，Level 2/3 子菜单是独立 DOM 节点不在 `ctxMenuRef` 内 → `mousedown` 时判定为"外部点击"立即关闭菜单 → `click` 事件永远触发不了
- **修复**：`contains` 检查增加三个条件（任一满足则不关闭）：
  1. `ctxMenuRef.current.contains(target)` — Level 1 菜单内
  2. `ctxSubmenuRef.current.contains(target)` — Level 2 子菜单内
  3. `target.closest('.v2-canvas-ctx-menu')` — 任意层级子菜单内（泛用覆盖）

## 外部剪贴板粘贴（Ctrl+V 创建源节点）

### 功能
当用户在应用外复制内容（文本/图片/视频/音频），Ctrl+V 粘贴到画布时，自动按内容类型创建对应的源节点。

### 新增 Store Action
- `pasteExternalContent(position)`: 通过 `navigator.clipboard.read()` 读取系统剪贴板
  - **图片** (image/png, image/jpeg, etc.) → `source-image` 节点（`role:'source'`, `type:'ai-image'`），`Blob` → `FileReader` → `dataUrl` → `imageUrl`
  - **视频** (video/mp4, video/webm, etc.) → `source-video` 节点（`role:'source'`, `type:'ai-video'`）
  - **音频** (audio/mpeg, audio/wav, etc.) → `source-audio` 节点（`role:'source'`, `type:'ai-audio'`）
  - **文本** (text/plain) → `source-text` 节点（`role:'source'`, `type:'ai-text'`），高度按行数自适应
  - 多条目粘贴：依次偏移 `+40,+40` 放置
  - 权限错误优雅处理：`NotAllowedError` → Toast "无剪贴板读取权限"
  - 新增 `blobToDataUrl()` 工具函数

### 触发路径
1. **Ctrl+V 快捷键**（`useKeyboardShortcuts.ts`）：内部剪贴板有内容 → `pasteNodes`（旧逻辑）；为空 → `dispatchEvent('canvas-paste-external')` → `Canvas.tsx` 监听事件 → 计算视口中心 → 调 `pasteExternalContent`
2. **右键菜单粘贴**（`useCanvasContextMenu.ts` → `CanvasContextMenu.tsx`）：粘贴按钮始终可点击（不再依赖 `clipboardLen`），内部剪贴板有内容 → `pasteNodes`；为空 → `pasteExternalContent`

### Bug 修复：复制图片/视频文件时提示"剪贴板无可识别内容"

**根因**：Windows 上从文件管理器复制文件时，系统剪贴板存的是 `text/uri-list`（文件路径 URI），不是 `image/png` / `video/mp4` 等二进制数据。原代码只检查 `image/*`、`video/*`、`audio/*`、`text/plain` 四种 MIME，导致 `text/uri-list` 走向"无法识别"分支。

浏览器复制图片时，剪贴板常只有 `text/html`（含 `<img>` 标签），也没有 `image/*`。

**修复**：在 `pasteExternalContent` 中新增两个 MIME 类型处理（优先级在 `text/plain` 之前）：

| MIME 类型 | 处理逻辑 |
|---|---|
| `text/html` | 解析 HTML，提取 `<img>` 的 `src`：`data:` URL 直接用；`file://` 走 Tauri FS 读取；`http(s)://` 用 fetch 下载 |
| `text/uri-list` | 解析 `file://` URI，按扩展名判定媒体类型（image/video/audio），通过 Tauri FS `readFile` 读取并转 dataUrl |

**新增基础设施**：
- `fileService.ts` 新增 `readFileToDataUrl(filePath)` 函数：Tauri 环境下用 `@tauri-apps/plugin-fs` 读本地文件；浏览器降级用 fetch。自动检测 MIME 类型并返回 `data:` URL。
- `getMimeType` 扩展支持 `.bmp`、`.svg`、`.avi`、`.mov`、`.mkv`、`.ogg`、`.flac`、`.aac`

**修改文件**：
| 文件 | 改动 |
|---|---|
| `services/fileService.ts` | +`readFileToDataUrl()` 导出；`getMimeType` 扩展更多扩展名 |
| `store/useAppStore.ts` | `pasteExternalContent` 新增 `text/html` 和 `text/uri-list` 分支 |

### Bug 修复 2：`navigator.clipboard.read()` 在 Windows WebView2 中仍无法识别剪贴板内容

**更深层根因**：`navigator.clipboard.read()` 在 Tauri WebView2 中权限受限，即使添加了 `text/html` / `text/uri-list` 分支，Windows 从文件管理器复制文件时，这些 MIME 类型也不一定出现在 `navigator.clipboard.read()` 的返回中。

**真正的可靠方案**：利用原生 `paste` 事件的 `DataTransfer`（`clipboardData`），它直接提供 `files` / `items` / `getData()`，不依赖 Clipboard API 权限。

**架构变更**：

| 之前 | 现在 |
|---|---|
| Ctrl+V → `dispatchEvent('canvas-paste-external')` → Canvas 监听自定义事件 → 调 `pasteExternalContent`（用 `navigator.clipboard.read()`） | Ctrl+V 内部为空时不阻止默认 → 原生 `paste` 事件触发 → Canvas 监听 `window.addEventListener('paste', ...)` → 调 `pasteExternalFromDataTransfer(e.clipboardData, pos)` |

**新增 Store Action**：`pasteExternalFromDataTransfer(dt, position)`
- 优先处理 `dt.files`（File 对象 — OS 文件复制最可靠）
- 回退处理 `dt.items`：`image/*` / `video/*` / `audio/*` → `getAsFile()`；`text/html` → `getData()` 提取 `<img>`；`text/uri-list` → `getData()` 解析文件路径；`text/plain` → 文本节点
- 使用"收集-执行"模式：先用 `getData()` 同步收集所有 pending 项，再统一异步执行，避免 `getAsString(callback)` 的时序问题
- 接口返回类型 `Promise<void>`

**修改文件**：
| 文件 | 改动 |
|---|---|
| `hooks/useKeyboardShortcuts.ts` | Ctrl+V 内部为空时不再 `preventDefault`，让原生 paste 事件自然触发 |
| `components/Canvas.tsx` | 移除 `canvas-paste-external` 自定义事件，改为 `window.addEventListener('paste', handler, true)` 捕获原生 paste 事件 |
| `store/useAppStore.ts` | 新增 `pasteExternalFromDataTransfer` action（~200 行） |
| `services/fileService.ts` | `readFileToDataUrl`（上一轮已添加，本轮复用） |

### Bug 修复：外部粘贴图片节点尺寸不按图片比例

**问题**：`addImageNode` / `pasteExternalContent` 中所有图片节点都使用固定 `nodeWidth: 280, nodeHeight: 158`，无视图片实际宽高比。

**修复**：新增 `computeImageNodeDimensions(dataUrl)` 工具函数（文件顶部），通过 `new Image()` 加载图片获取 `naturalWidth`/`naturalHeight`，计算比例尺寸：
- `nodeWidth` clamp 到 160~280 范围
- `nodeHeight = Math.round((nodeWidth - 4) / ratio + 4)`，最小 120
- 图片加载失败时回退到 280×158

所有 4 处图片节点创建点均已改用 `computeImageNodeDimensions`：
- `pasteExternalFromDataTransfer` 的 `addImageNode` helper → 改为 async，用 `...dims`
- `pasteExternalContent` 的 3 个分支（image MIME、text/html、text/uri-list）

**修改文件**：`store/useAppStore.ts`
