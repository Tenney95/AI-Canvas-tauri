## Bugfix: workflow chip 内/外 @ 画布节点 chip 的边界识别

- **问题**：`@wf{id:title:type}` 格式没有明确的 value 区边界，导致无法区分嵌套在 value 区内的 `@{canvasId}` 和顶层 sibling `@{canvasId}`
- **根因**：`serializeDOM` 输出 `@wf{id:title:type}@{canvasId}` 时，value 区内的 chip 和顶层 chip 在字符串层面完全一致，`renderPromptToNodes` 和 `AINodeDialog.onPromptChange` 都会误判
- **修复（3 个文件）**：
  - **`serializeDOM`**：value 区内容用 `(...)` 包裹，输出 `@wf{id:title:type}(value @{canvasId:label})`
  - **`renderPromptToNodes`**：匹配 `@wf{...}` 后检查紧跟 `(`，用深度追踪找匹配 `)`，递归渲染 value 区内的 `@{...}` chip
  - **`AINodeDialog.onPromptChange`**：正则改为 `/@wf\{([^:]+):([^:]+):([^}]+)\}\(([\s\S]*?)\)/g` 精准提取每个 workflow IO 节点的 value
- **格式**：`@wf{ioNodeId:title:type}(...nested content...)` — 顶层 `@{...}` 不受影响，仅在 `(...)` 内的 `@{...}` 被当作嵌套 chip

## Bugfix: 点击 @ 菜单项时 selection 丢失导致 chip 插入位置错误

- **问题**：在 workflow chip 的 value 区内输入 `@` 后，点击菜单项时编辑器失去焦点，`window.getSelection()` 的 range 已丢失，导致 `insertChipAtCursor` 将 chip 插入到编辑器末尾而非 value 区内部
- **修复**（`MentionEditor.tsx`）：
  - 新增 `savedMentionRangeRef` — 在 `handleInput` 检测到 `@` 时 `cloneRange()` 保存当前光标位置
  - `handleSelectCanvasMention` / `handleSelectWorkflowMention` — 插入前先恢复 saved range 到 `window.getSelection()`
  - 新增 `useEffect` — `showMention` 变为 false 时清除 saved range（处理 Escape、点击外部等关闭场景）

## Bugfix #3: `el.focus()` 在恢复 saved range 后导致 selection 被重置

- **问题**：修复 #2 后仍然不行 — canvas chip 依旧出现在 workflow chip 外部而非内部
- **根因**：`handleSelectCanvasMention` 先恢复 saved range（指向 `.prompt-chip-wf-value` 内的文本节点），然后 `insertChipAtCursor` 调用 `el.focus()`。由于 saved range 的 startContainer 与主编辑器之间隔着 `contentEditable="false"` 的 workflow chip，浏览器在 `focus()` 时将 selection 重置到主编辑器顶层，导致 `range.insertNode(chip)` 插入到错误位置
- **修复**（`MentionEditor.tsx`）：
  - `handleSelectCanvasMention` / `handleSelectWorkflowMention`：**先 `el.focus()` 再恢复 saved range**（在 focus 状态下 set selection 能跨 contentEditable 边界正确设置）
  - `deleteAtChar`：移除无用的 `const el = editorRef.current`（不做 focus 操作了）
  - `insertChipAtCursor` / `insertWorkflowChipAtCursor`：移除 `el.focus()` 调用（不再需要，靠 handler 里先 focus）

## Bugfix #4: workflow chip value 区每次序列化/反序列化后多出 `<br>`

- **问题**：在 workflow chip value 区内插入 canvas chip 后，关闭再打开对话框，value 区内多出一个 `<br>` 标签
- **根因**：`buildWorkflowChipEl` 在 `.prompt-chip-wf-value` 内预置 `<br>` 占位符（保持空 contentEditable 可编辑），但 `renderPromptToNodes` 追加嵌套内容时没有先清空 value 区，导致 `<br>` + 嵌套内容共存
- **修复**（`MentionEditor.tsx` 第 178 行）：在 `renderPromptToNodes` 中追加嵌套 nodes 到 value 区之前，先 `valueArea.innerHTML = ''` 清除初始占位符
