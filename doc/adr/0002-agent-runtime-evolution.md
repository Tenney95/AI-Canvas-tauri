# ADR 0002：Agent 运行时调度、恢复与扩展边界

## 状态

已接受，2026-07-21。

## 背景

P3 Agent 已具备 Tool Registry、Policy Engine、多轮循环、审批、后台任务和重启暂停，但同会话可并发启动多个任务，恢复时缺少已完成步骤上下文，也没有任务级安全回退和隐私友好的诊断数据。后续还需要相关性记忆、结构化 Skill、只规划模式和只读专家任务。

## 决策

1. 同一会话的主任务使用进程内 FIFO 调度；任务本身继续由 `AgentTask` 持久化。运行时队列不持久化，重启后统一恢复为暂停。
2. 事件日志、指标、父子关系和检查点作为 `AgentTask` 的可选字段嵌入现有 `agentTasks` store，不新增 IndexedDB object store。事件数量有上限，数据字段使用白名单。
3. 恢复时只注入已完成步骤的脱敏摘要。相同工具 ID 与输入哈希的成功写调用返回既有 Observation，不重新执行。
4. 任务回退只使用现有 Store `undo()`，且仅在检查点链仍是当前历史连续尾部、projectId 和 revision 都匹配时执行。回退后 revision 单调递增，不恢复旧 revision。
5. Skill Manifest 的工具声明只能缩小 Registry 可见集合。`plan` 模式同时通过 Registry 过滤与 Policy 拒绝保护写操作。
6. 专家任务深度固定为 1，使用独立无工具模型轮次。专家只能接收脱敏画布结构，不能获得写工具、媒体生成、文件路径或任意网络能力。
7. 生命周期扩展只提供进程内类型化事件。外部 Shell、HTTP Hook、MCP 和插件市场不在本阶段范围内，任何监听器都不能改变 Policy 结果。

## 影响

- 旧任务、会话和 Skill 通过 normalize 默认值继续读取，不需要 DB_VERSION 迁移。
- 同会话吞吐变为串行，但消除了消息顺序、审批归属和 revision 竞争。
- 事件和指标增加任务记录体积，因此事件上限固定为 200，并禁止持久化正文。
- 任务整体回退是保守能力；发生任何交错写入时宁可拒绝，也不覆盖用户修改。

## 回滚

各阶段通过移除 UI 入口和对应 Runtime 装配回滚。新增字段保持可选，旧代码可忽略；不得降低 IndexedDB 版本或删除现有任务。Policy Engine、安全矩阵和 Tauri capability 不随回滚变化。
