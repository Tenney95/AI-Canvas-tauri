# 打包与发版流程

> 适用于已启用 `createUpdaterArtifacts`、自动更新和 GitHub Actions 质量门禁的版本。
> 当前版本以 `package.json` 为唯一来源，不在本文档中固定具体版本号。

## 前置准备

### 需要的文件 / 配置

| 文件 | 作用 |
|---|---|
| `package.json` → `version` | 前端版本号（唯一来源） |
| `src-tauri/tauri.conf.json` → `version` | Tauri 版本号，运行时读取 |
| `src-tauri/Cargo.toml` → `package.version` | Rust 编译版本号 |
| `src-tauri/update-key.key` | 签名私钥（**敏感，勿提交**） |
| `src-tauri/update-key.key.pub` | 签名公钥（已写入 `tauri.conf.json` 的 `plugins.updater.pubkey`） |
| `src-tauri/tauri.conf.json` → `bundle.createUpdaterArtifacts` | 设为 `true`，构建时生成 `.sig` 签名文件 |
| `.github/workflows/ci.yml` | PR 与 `master` push 的前端质量检查和 Rust 检查 |
| `.github/workflows/release.yml` | Tag 发布、质量前置检查与三平台安装包构建 |

### GitHub Secrets（无需每次修改）

在仓库 **Settings → Secrets and variables → Actions** 中配置：

| Secret | 值 |
|---|---|
| `TAURI_SIGNING_PRIVATE_KEY` | `update-key.key` 文件内容（一行 `dW50cnVzdGVk...`） |
| `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` | 生成密钥时设置的密码 |

> 密钥通过 `npx tauri signer generate -w ./src-tauri/update-key.key -f -p "<签名密码>" --ci` 生成。
> 每次重新生成密钥后，必须连同 `tauri.conf.json` 中的 `pubkey` 一起更新。
> 私钥、签名密码和包含它们的终端输出不得提交到仓库或写入发版记录。

## 第一步：版本号同步

### 1.1 修改 `package.json` 的 `version`

例如将当前版本 `X.Y.Z` 更新为目标版本，并确保符合语义化版本规则。

### 1.2 运行同步脚本

```powershell
npm run sync-version
```

这会自动同步版本号到：
- `src-tauri/Cargo.toml`（Rust 编译时不再显示旧版本）
- `README.md`（版本 badge）

### 1.3 手动修改 `src-tauri/tauri.conf.json`

```json
"version": "X.Y.Z"
```

> 三处版本号（`package.json`、`tauri.conf.json`、`Cargo.toml`）必须一致。

---

## 第二步：质量门禁

质量检查必须在创建 Tag 前完成。不得在检查失败时继续发布。

### 2.1 前端检查

```powershell
npm ci
npm run check
```

`npm run check` 当前依次执行：

- 前端生产代码类型检查；
- 测试代码类型检查；
- Vitest 自动化测试。

CI 使用 `npm run ci`，在上述检查之后继续执行前端生产构建。

> `npm run lint` 当前受 ESLint 10.4 与 TypeScript parser 的既有兼容问题阻断，错误为
> `scopeManager.addGlobals is not a function`。在工具链专项修复前，lint 不属于必过 CI job；
> 不得通过降级或替换依赖掩盖该错误。

### 2.2 Rust 检查

```powershell
Set-Location src-tauri
cargo check --lib
Set-Location ..
```

### 2.3 CI 检查

推送到 `master` 或创建 Pull Request 后，`.github/workflows/ci.yml` 自动执行：

1. Ubuntu 前端 job：`npm ci` → `npm run ci`；
2. Windows Rust job：`cargo check --lib`。

两个 job 都通过后，代码才具备创建发布 Tag 的条件。

## 第三步：本地构建

### 3.1 设置环境变量

```powershell
$env:TAURI_SIGNING_PRIVATE_KEY = (Get-Content "src-tauri/update-key.key" -Raw).Trim()
$env:TAURI_SIGNING_PRIVATE_KEY_PASSWORD = "<签名密码>"
```

> `Get-Content -Raw` 读取文件内容，`.Trim()` 去除尾随换行符，防止 base64 污染。
> `<签名密码>` 替换为生成密钥时设置的密码，不要把真实密码写入脚本或文档。

### 3.2 构建

```powershell
npm run tauri build
```

构建完成后产物位置：

```
src-tauri/target/release/bundle/
├── nsis/
│   ├── AI Canvas_X.Y.Z_x64-setup.exe      ← 安装包
│   └── AI Canvas_X.Y.Z_x64-setup.exe.sig  ← 签名文件
├── msi/
│   ├── AI Canvas_X.Y.Z_x64_en-US.msi
│   └── AI Canvas_X.Y.Z_x64_en-US.msi.sig
└── ...
```

### 3.3 检查产物

确认以下文件均已生成：
- `.exe` / `.msi` 安装包
- 对应的 `.sig` 签名文件
- 终端无报错

---

## 第四步：创建 `latest.json`

Tauri 更新器通过拉取 `latest.json` 来判断是否有新版本。本地构建不会自动生成此文件。

### 4.1 模板

```json
{
  "version": "X.Y.Z",
  "notes": "本版更新说明……",
  "pub_date": "2026-07-06T11:30:00Z",
  "platforms": {
    "windows-x86_64": {
      "signature": "<把 .sig 文件内容贴到这里>",
      "url": "https://github.com/Tenney95/AI-Canvas-tauri/releases/download/vX.Y.Z/AI.Canvas_X.Y.Z_x64-setup.exe"
    }
  }
}
```

- `signature`：复制 `AI Canvas_X.Y.Z_x64-setup.exe.sig` 文件里的全部内容
- `url`：GitHub Release 的直接下载链接。注意 GitHub 会把文件名中的空格替换为 `.`
- `pub_date`：当前 UTC 时间，格式 `YYYY-MM-DDTHH:mm:ssZ`

### 4.2 上传方式

**通过 CI 自动发布（推荐）**：跳过此步，`tauri-action` 的 `includeUpdaterJson: true` 会自动生成并上传。

**手动发布**：需自行创建 `latest.json` 并上传到 Release Assets。

---

## 第五步：发布 GitHub Release

### 方式 A：自动发布（推荐）

```powershell
git status --short
git add <本次发版文件>
git commit -m "chore(release): 发布 vX.Y.Z"
git push origin master

# 等待 master 上的 CI 全部通过后再创建 Tag
git tag vX.Y.Z
git push origin vX.Y.Z
```

推送 tag 后，`.github/workflows/release.yml` 自动触发：

1. `quality` job 执行 `npm ci` 和 `npm run ci`；
2. 只有 `quality` 通过后，`publish` job 才开始构建 Windows / macOS / Linux 安装包；
3. 生成 `.sig` 签名文件；
4. 生成 `latest.json`；
5. 创建 GitHub Release 并上传所有 assets。

> 发布 Tag 应视为不可变。Tag 推送后发现问题时，修复代码并发布新补丁版本，禁止覆盖已经发布的 Tag。

### 方式 B：手动发布

1. 打开 https://github.com/Tenney95/AI-Canvas-tauri/releases/new
2. Tag 填 `vX.Y.Z`，Release title 填 `AI Canvas vX.Y.Z`
3. 上传以下文件到 Assets：
   - `AI Canvas_X.Y.Z_x64-setup.exe`（安装包）
   - `AI Canvas_X.Y.Z_x64-setup.exe.sig`（签名）
   - `latest.json`（版本描述）
4. 填写 Release notes，点击 **Publish release**

> 手动发布时必须上传 `latest.json`，否则自动更新会报 `Could not fetch a valid release JSON`。

## 第六步：验证更新

### 6.1 本地验证

1. 安装目标版本的上一版本
2. 重新构建安装包并安装
3. 启动应用 → 观察控制台 → 应自动拉取 `latest.json` 并开始下载新版本
4. 下载完成后应用自动重启至新版本

### 6.2 验证要点

- 控制台输出 `[Updater] 发现新版本 vX.Y.Z，开始下载...`
- 进度条显示下载进度
- 下载完成后自动安装并重启
- 重启后版本号显示为新版本
- 如果在已是最新版的机器上打开，输出 `当前已是最新版本`

### 6.3 常见问题

| 现象 | 原因 | 解决 |
|---|---|---|
| `开发模式，跳过更新检查` | `npm run tauri dev` 运行 | 正常，打包后自动不跳过 |
| `Could not fetch a valid release JSON` | `latest.json` 未上传到 Release Assets | 手动上传 |
| `pubkey is not valid` / 签名验证失败 | 本地构建用的 key 与 GitHub Release 的签名不匹配 | 确保用同一密钥对 |
| `failed to decode secret key` | 环境变量未设置或 base64 含换行符 | `Get-Content -Raw` 后加 `.Trim()`，并设置密码 |
| `failed to decode base64 secret key` | `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` 未设置 | 检查密码是否与生成密钥时一致 |
| Release 的 `publish` 未启动 | `quality` job 未通过 | 查看 `npm run ci` 的类型、测试或构建失败日志，修复后发布新 Tag |
| CI 中 lint 未执行 | ESLint 10.4/parser 兼容问题尚未修复 | 这是已知工具链阻塞；单独修复后再恢复 lint 门禁 |

```
        │

```
