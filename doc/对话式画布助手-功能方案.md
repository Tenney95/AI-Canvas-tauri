# 如何用对话安全地读取和操作 AI 画布

> 文档类型：功能与技术方案
> 版本：v0.7
> 日期：2026-07-11
> 适用项目：AI-Canvas-tauri

本方案定义 AI 驱动的侧边对话助手，包括多会话、历史管理、流式回答、联网搜索、本地文件工具和多媒体生成。图像、视频、音乐和语音生成支持聊天呈现、画布节点或双端呈现；模型调用与节点物化彼此独立，并共享同一生成任务和产物。

## 0. 文档目标与受众

本文帮助产品、设计和工程协作者确认以下事项：

- 助手能理解哪些画布指令
- AI 响应如何以事件流增量传输和渲染
- 多个会话如何创建、切换、搜索、归档和删除
- 对话历史如何持久化、分页、恢复和清理
- 联网搜索如何返回可追溯来源并防止任意网络访问
- 本地文件如何按用户授权读取、搜索、导入和导出
- 图像、视频、音乐和语音如何在对话中生成并呈现
- 同一生成结果如何按需添加到画布而不重复调用模型
- 指令如何解析、预览、确认、执行和撤销
- 哪些能力可以复用，哪些能力需要先抽象
- 用户数据何时会发送给外部模型
- 每个阶段如何验收和回滚

本文不提供最终代码实现，也不授权直接开展超过 3 个文件的代码修改。进入实施前，仍需按 `AGENTS.md` 展示文件清单、影响面和回滚方案并等待确认。

## 1. 背景、目标与范围

AI-Canvas 是基于 Tauri 2、React 19、React Flow 12 和 Zustand 5 的节点式 AI 创作画布。当前操作主要依赖鼠标和快捷键，缺少用自然语言完成查询、多选和批量操作的入口。

对话助手是由 AI 驱动的画布操作增强层，不是新的媒体生成引擎。它承担三类任务：

- **对话理解**：结合画布摘要理解自然语言、指代和上下文
- **流式回答**：逐步展示解释、澄清和查询结果，允许用户随时停止
- **操作画布**：把明确指令转换为受控的 Store Action，并提供预览、确认、结果和撤销能力

### 1.1 成功标准

本方案完整落地后的可用版本需要达到以下结果；各项能力仍按后文阶段逐步交付：

- 助手使用明确配置的 AI 模型理解用户输入
- 模型响应通过统一事件流增量展示，用户可以中止请求
- 用户可以在同一项目内创建和切换多个独立会话
- 会话和消息在应用重启后仍可恢复
- 用户可以重命名、置顶、归档、移入回收站、恢复和永久删除会话
- 不同会话的 AI 上下文不会混用
- 用户可以要求助手联网搜索，并查看每条结论的来源
- 用户可以通过文件选择器授权本地文件或文件夹
- 文件授权只在指定项目和会话中生效，模型看不到任意路径
- 用户可以只在对话中生成媒体，不创建任何画布节点
- 用户可以直接生成画布节点，或让聊天与画布共享同一产物
- 用户可以查询画布状态并定位节点
- 用户可以删除失败节点，并在一次 `Ctrl+Z` 中完整恢复
- 助手不会因模糊指令执行写操作
- 模型输出不能绕开本地命令白名单和确认策略
- 每条命令都能解释目标、参数、风险和执行结果

### 1.2 明确不做

以下能力不属于首个版本：

- 任意脚本、Shell 命令或动态代码执行
- 绕过 Store Action 直接修改 React Flow 状态
- 未经许可读取画布外文件或把媒体发送给外部服务
- 由 LLM 直接生成 Store patch、节点 ID 或确认结论
- 在统一生成运行时完成前，通过模拟组件点击触发重生成
- 假定所有远端生成任务都支持取消
- 跨项目共享同一会话上下文
- 把删除本地会话描述为删除供应商侧请求记录
- 任意 URL 请求、内网探测或自动跟随不受控重定向
- 未经用户选择读取磁盘、遍历主目录或持久化全盘权限
- 删除、移动、执行本地文件或覆盖原文件

## 3. 总体架构

助手采用“AI-first、流式输出、确定性执行”的管线。AI 负责理解和对话，本地规则负责无模型或断网降级；两者只能提出意图，本地规划器负责目标解析、参数校验、风险评估和确认策略。

```text
用户输入
  → 当前项目与活动会话
  → 脱敏画布上下文
  → AI Stream Adapter
  → text.delta / tool.call.final / intent.final / done
  → Tool Registry 授权与执行
  → 脱敏 ToolResult 返回模型
  → 完整 CommandIntent
  → 目标选择器 Selector AST
  → 基于当前项目快照生成 CommandPlan
  → 本地 schema 与策略校验
  → 预览和必要确认
  → Store 事务或只读查询
  → CommandResult 与操作日志
```

### 3.1 组件职责

| 层 | 职责 | 禁止事项 |
|---|---|---|
| 对话输入层 | 输入、提及、队列、快捷键和状态展示 | 不直接调用 Store 写 Action |
| 会话管理层 | 创建、切换、命名、归档、删除和恢复会话 | 不跨会话拼接上下文 |
| 历史存储层 | 分页读取消息、增量落盘和执行 schema 迁移 | 不把全部历史常驻内存 |
| AI 对话层 | 组装脱敏上下文并请求配置的助手模型 | 不发送未授权数据 |
| 流适配层 | 把 SSE、NDJSON 或供应商分片统一为事件流 | 不把半截 JSON 当成最终意图 |
| 意图解析层 | 从完整模型结果提取命令 ID、Selector AST 和候选参数 | 不返回最终节点 ID，不决定是否确认 |
| 本地降级层 | 在模型不可用时识别核心命令 | 不假装已经使用 AI 回答 |
| 工具注册表 | 定义联网、文件工具的 schema、权限和执行器 | 不接受模型提供的任意 URL 或路径 |
| 授权层 | 管理会话级文件 grant、联网许可和确认 | 不把应用级 Tauri 权限直接授予模型 |
| 规划层 | 在画布快照上解析目标、计算差异和风险 | 不产生副作用 |
| 命令注册表 | 定义参数 schema、目标约束、风险和执行函数 | 不接受未注册字段或命令 |
| 策略层 | 决定确认、批量上限、数据发送许可和能力降级 | 不信任 LLM 的安全结论 |
| 执行层 | 调用 Store 事务、查询服务或生成运行时 | 不在组件中散落命令分支 |
| 反馈层 | 展示结果、错误、可撤销状态和诊断信息 | 不把“停止跟踪”描述为“取消远端任务” |

### 3.2 核心数据结构

意图只描述用户想做什么，不携带最终节点 ID 或确认结果。

```typescript
type CommandId =
  | 'query'
  | 'select'
  | 'deleteNodes'
  | 'undo'
  | 'redo'
  | 'connect'
  | 'groupByType'
  | 'translatePrompt'
  | 'regenerate'
  | 'describe'
  | 'cancelTask';

interface CommandIntent {
  commandId: CommandId;
  selector?: NodeSelector;
  params?: unknown;
  parseSource: 'rule' | 'llm';
  confidence: number;
}
```

Selector AST 表达交集、并集和排除关系，避免把所有实体一律取并集。

```typescript
type NodeSelector =
  | { op: 'selected' }
  | { op: 'displayId'; value: number }
  | { op: 'type'; value: NodeType }
  | { op: 'status'; value: NodeStatus }
  | { op: 'and'; items: NodeSelector[] }
  | { op: 'or'; items: NodeSelector[] }
  | { op: 'not'; item: NodeSelector };
```

规划器解析 Selector 后生成不可变预览。执行前会再次检查项目和画布版本。

```typescript
interface CommandPlan<TParams> {
  id: string;
  projectId: string;
  baseRevision: number;
  commandId: CommandId;
  targetNodeIds: string[];
  params: TParams;
  summary: string;
  risk: 'read' | 'low' | 'destructive' | 'external';
  requiresConfirm: boolean;
  externalData?: ExternalDataDisclosure;
}

interface CommandResult {
  planId: string;
  status: 'success' | 'partial' | 'rejected' | 'failed';
  affectedNodeIds: string[];
  message: string;
  errorCode?: string;
  historyIndex?: number;
}
```

## 4. AI 对话、流式协议与目标选择

P0-A 默认调用 AI 模型理解输入并生成流式回答。模型不可用、用户主动关闭云端解析或网络失败时，本地规则只降级处理查询、选择、删除、撤销和重做等核心命令。

### 4.1 AI-first 意图解析

模型同时生成面向用户的文本和结构化意图。文本可以增量展示；意图必须完整结束并通过本地 schema 校验后，才能交给规划器。

```json
{
  "intents": [
    {
      "commandId": "deleteNodes",
      "selector": {
        "op": "status",
        "value": "error"
      },
      "params": {},
      "confidence": 0.94
    }
  ]
}
```

模型不能返回真实节点 ID、`requiresConfirm`、Store patch 或执行结果。面向用户的解释也不能代替本地 `CommandResult`。

### 4.2 脱敏画布上下文

AI 上下文默认只包含以下信息：

- 当前项目的临时节点别名，例如 `n1`、`n2`
- 节点类型、状态、`displayId` 和是否选中
- 聚合后的节点和边数量
- 当前支持的命令和 Selector schema
- 本轮对话所需的有限历史消息

上下文默认不包含节点 label、prompt、output、媒体 URL、本地路径和 API Key。节点 label 可能包含隐私信息或提示注入内容，只有在用户明确允许后才能发送。

### 4.3 统一流式事件协议

供应商 Adapter 把 Server-Sent Events（SSE）、换行分隔 JSON（NDJSON）或其他分片格式归一化为 `AsyncIterable<AssistantStreamEvent>`。

```typescript
type StreamPhase = 'connecting' | 'responding' | 'planning';
type FinishReason = 'stop' | 'length' | 'canceled' | 'error';

type AssistantStreamEvent =
  | { type: 'start'; requestId: string; modelId: string }
  | { type: 'text.delta'; delta: string }
  | { type: 'status'; phase: StreamPhase; message?: string }
  | { type: 'conversation.title'; title: string }
  | { type: 'tool.call.delta'; callId: string; delta: string }
  | { type: 'tool.call.final'; call: ProposedToolCall }
  | { type: 'tool.result'; result: ToolResultSummary }
  | { type: 'intent.final'; intents: CommandIntent[] }
  | { type: 'usage'; inputTokens?: number; outputTokens?: number }
  | { type: 'error'; code: string; message: string; retryable: boolean }
  | { type: 'done'; finishReason: FinishReason };
```

事件语义如下：

- `text.delta` 只更新消息展示内容，不触发命令
- `status` 展示“正在分析画布”“正在生成计划”等阶段信息
- `conversation.title` 只在首轮且标题未被手动修改时更新会话标题
- `tool.call.delta` 只用于展示工具参数正在生成，不能执行
- `tool.call.final` 进入本地 Tool Registry 校验、授权和确认
- `tool.result` 只携带经过裁剪和脱敏的工具结果摘要
- `intent.final` 是唯一可进入本地规划器的模型事件
- `usage` 用于费用提示和诊断，不参与执行判断
- `done` 关闭流并固化消息状态
- `error` 保留已收到文本，但清除未完成意图

### 4.4 流式传输链路

P0-A 新增独立的助手模型接口，不复用当前非流式 `generateText`：

```typescript
function streamAssistantResponse(
  request: AssistantRequest,
  options: { signal: AbortSignal },
): AsyncIterable<AssistantStreamEvent>;
```

传输层按运行环境选择：

- 前端可以直连的 OpenAI-compatible 服务使用 `fetch()` 和 `ReadableStream`
- 必须经 Rust 代理的服务使用 Tauri IPC Channel 持续推送分片
- 两种传输都先转换为 `AssistantStreamEvent`，ChatPanel 不感知底层来源

Tauri 2 官方文档把 Channel 作为流式 HTTP 响应等数据的推荐机制，不能用等待完整 Promise 的普通 `invoke()` 代替流传输。参见 [Tauri Channels](https://v2.tauri.app/develop/calling-rust/#channels)。

该接口需要支持：

- system 与 user 消息分离
- `stream: true` 和响应体增量读取
- UTF-8 多字节字符跨 chunk 拼接
- SSE 与 NDJSON 分片解析
- `AbortSignal` 主动停止
- 明确的 `assistantModelId`
- `streaming`、`structuredOutput` 和 `vision` 能力标记
- 首字节、空闲和总时长三类超时
- 禁止自动重试付费请求

Chat Store 不应为每个 token 触发一次 React 更新。流消费者先缓冲 `text.delta`，再每 50 ms 或每个动画帧合并提交一次。

### 4.5 中止、背压与断流恢复

流式请求必须提供停止按钮，并为每个请求维护独立 `AbortController`。处理规则如下：

| 场景 | 处理 |
|---|---|
| 用户停止 | 中止网络读取，保留已显示文本，状态改为 canceled |
| 首字节超时 | 取消请求，允许本地规则处理核心命令 |
| 流空闲超时 | 标记 interrupted，不执行未完成意图 |
| 供应商断流 | 保留文本并显示“响应中断”，不自动付费重试 |
| 收到未知事件 | 忽略并记录诊断，不影响其他事件 |
| 缓冲超过上限 | 中止请求并返回 `STREAM_LIMIT_EXCEEDED` |
| 项目切换 | 取消旧项目的活动流和未确认计划 |

只有同时收到合法 `intent.final` 和正常 `done` 的请求才能生成写命令预览。只读自然语言回答可以保留部分文本，但必须标记为未完成。

### 4.6 本地规则降级

本地规则遵循最长匹配优先，并输出与 AI 相同的 `CommandIntent`。它不是主要理解引擎，只在模型未配置、云端解析关闭或请求失败时处理核心命令。

| 命令 ID | 触发表达 | 目标要求 | P0-A 降级 |
|---|---|---|---|
| `query` | 查、统计、几个、哪些、多少 | 可选 | 是 |
| `select` | 选中、选择、定位到 | 必填 | 是 |
| `deleteNodes` | 删除、删掉、移除、清除 | 必填 | 是 |
| `undo` | 撤销、回退、退一步 | 无 | 是 |
| `redo` | 重做、恢复重做 | 无 | 是 |

目标组合仍由本地 Selector 解析：

- 「失败的图片节点」解析为 `and(status=error, type=ai-image)`
- 「图片和视频节点」解析为 `or(type=ai-image, type=ai-video)`
- 「失败节点，不包括文本节点」解析为 `and(status=error, not(type=ai-text))`
- 「第 3 个节点」按 `data.displayId === 3` 解析，不使用数组下标
- 单数指代只在恰好选中一个节点时解析为 `selected`

### 4.7 多意图、置信度与执行权限

P0-A 可以流式解释多个意图，但多个写命令仍需逐条确认。置信度只决定继续规划或反问，不决定确认策略。

1. AI 或本地规则生成完整意图
2. 本地 schema 校验命令和 Selector
3. 每个意图生成独立计划
4. 每个计划执行前检查最新 `baseRevision`
5. 前一条失败时暂停后续命令

低置信度、缺少目标或存在多个解释时，助手流式返回澄清问题，不生成可执行计划。

## 5. 命令注册表与参数校验

命令注册表是唯一执行入口。每条命令定义自己的参数 schema、目标约束、风险级别、预览函数和执行函数。

```typescript
interface CommandDefinition<TParams> {
  id: CommandId;
  category: 'read' | 'canvas-write' | 'external-effect';
  validateParams: (input: unknown) => TParams;
  validateTargets: (ctx: PlanContext, ids: string[]) => void;
  plan: (ctx: PlanContext, intent: CommandIntent) => CommandPlan<TParams>;
  execute: (ctx: ExecuteContext, plan: CommandPlan<TParams>) => Promise<CommandResult>;
}
```

`params` 不能使用无约束的 `Record<string, unknown>` 后直接展开到 `updateNodeData`。每条命令必须只接受白名单字段，例如：

```typescript
interface DeleteNodesParams {
  includeDescendants: boolean;
}

interface ConnectParams {
  strategy: 'pairwise' | 'all-to-one';
  sourceHandle?: string;
  targetHandle?: string;
}

interface RegenerateParams {
  aspectRatio?: string;
  imageSize?: string;
}
```

参数校验可以手写，也可以在用户确认新增依赖后引入 Zod。P0-A 默认采用手写类型守卫，避免把依赖决策混入首个功能切片。

## 6. Store 事务与撤销语义

当前 `deleteNode`、`updateNodeData`、`onConnect` 和分组 Action 会各自提交历史。循环调用这些 Action 无法实现“一个批次只产生一条历史”。

P0-A 必须先提供批量事务能力。推荐新增专用原子 Action，而不是给所有 Action 增加可选的“跳过历史”参数：

```typescript
interface DeleteNodesBatchInput {
  nodeIds: string[];
  expectedRevision: number;
}

interface DeleteNodesBatchResult {
  deletedNodeIds: string[];
  skippedNodeIds: string[];
  historyIndex: number;
}
```

批量 Action 需要满足以下要求：

1. 验证当前项目和 `expectedRevision`
2. 展开并去重子节点
3. 计算受影响的边、组和本地文件
4. 只提交一次历史快照
5. 一次性更新节点、边和组
6. 异步清理文件并记录失败结果
7. 返回稳定的 `CommandResult`

### 6.1 原子与部分成功策略

画布状态变更采用原子策略：校验失败时不修改 Store。文件清理和远端调用属于外部副作用，不能与内存状态形成真正的跨系统事务，因此需要单独报告。

| 结果 | Store 状态 | 外部副作用 | 用户反馈 |
|---|---|---|---|
| 校验失败 | 不变 | 不执行 | 显示失败原因 |
| Store 更新成功 | 全部提交 | 继续处理 | 显示成功和撤销入口 |
| 文件清理部分失败 | 保持已提交状态 | 记录失败文件 | 显示部分成功，可撤销画布状态 |
| 远端调用失败 | 按命令策略恢复或标错 | 不自动重试 | 显示供应商错误 |

### 6.2 操作日志中的撤销

现有历史是线性栈，因此操作日志只允许撤销当前最新且可撤销的画布命令。旧日志只显示详情，不提供“撤销此步”。未来如需撤销中间步骤，必须在以下方案中另行决策：

- 回退到该历史点并丢弃其后的历史
- 为每条命令实现补偿命令
- 引入更复杂的分支历史

P0 不实现操作重放。日志中的“查看计划”只展示当时的目标和结果，不会重新执行命令。

## 7. 各命令的执行约束

命令表明确区分当前可复用能力、需要包装的能力和前置架构工作。

| 命令 | 本地实现 | 风险 | 确认策略 | 阶段 |
|---|---|---|---|---|
| `query` | 读取 `useAppStore.getState()` | read | 无 | P0-A |
| `select` | `setSelectedNodeIds()` | low | 无 | P0-A |
| `deleteNodes` | 新增原子批量 Action | destructive | 始终确认 | P0-A |
| `undo` / `redo` | `await undo()` / `await redo()` | low | 无 | P0-A |
| `connect` | 新增去重和批量包装 | destructive | 新增多条边时确认 | P0-B |
| `groupByType` | 新增按类型批量分组 Action | destructive | 影响多个节点时确认 | P0-B |
| `translatePrompt` | 助手模型调用后显示差异 | external | 发送数据和写回前确认 | P2-C |
| `regenerate` | 统一生成运行时 | external | 展示模型、参数和费用风险 | P2-C |
| `describe` | 视觉模型调用 | external | 发送媒体前确认 | P2-C |
| `cancelTask` | 供应商能力适配 | external | 明确说明取消语义 | P2-C |

### 7.1 连线约束

`connect` 不能直接循环调用现有 `onConnect`。规划器需要先计算合法边，并应用以下限制：

- 拒绝自连
- 跳过相同 source、target 和 handle 的重复边
- 校验来源和目标 handle 的兼容性
- 默认不创建循环依赖；未来允许时需单独配置
- 多来源和多目标时必须明确 `pairwise` 或 `all-to-one`
- 默认最多新增 100 条边；超限时拒绝并要求缩小范围
- 执行前展示新增、跳过和拒绝的边数量

### 7.2 按类型分组约束

现有 `groupSelectedNodes` 只把当前选择创建为一个组，不能直接实现“按类型分组”。`groupByType` 需要新的批量 Action，并遵循以下规则：

- 按 `data.type` 建立独立分组
- 忽略少于 2 个节点的类型
- 已在组内的节点默认跳过
- 一次命令只提交一条历史
- 预览需要列出将创建的组和节点数量

### 7.3 重生成前置条件

当前生成流程分布在节点组件和 `AINodeDialog.tsx`。P2 接入 `regenerate` 前，需要抽出统一入口：

```typescript
executeNodeGeneration(
  nodeId: string,
  overrides: RegenerateParams,
  options: { signal: AbortSignal },
): Promise<GenerationResult>
```

该入口必须复用现有 provider 和 workflow 分流，不能在助手中新增平行的模型判断。参数映射需要区分 `aspectRatio`、`seedanceRatio` 等不同模型字段。

### 7.4 取消任务语义

当前 `cancelNodePolling` 只停止本地轮询并清理待续记录。P2 需要为每个 provider 声明能力：

| 能力 | 用户文案 | 状态处理 |
|---|---|---|
| `cancelRemoteTask` | 已请求供应商取消任务 | 等待供应商确认结果 |
| `stopTrackingOnly` | 已停止在本机跟踪；远端任务可能继续 | 节点标记为停止跟踪 |
| `notCancelable` | 当前供应商不支持取消 | 保持任务状态 |

助手不得把 `stopTrackingOnly` 描述为远端取消成功，也不得承诺退还费用。

## 8. 安全、隐私与数据发送

安全策略全部由本地代码计算。模型返回值、节点标签、提示词、输出内容和媒体都属于不可信输入。

### 8.1 本地安全规则

- 写命令必须包含明确动词和可解析目标
- 未注册命令直接拒绝
- 每条命令使用独立参数 schema
- 计划中的 `projectId` 必须匹配当前项目
- 计划中的 `baseRevision` 过期时重新规划并再次预览
- 删除类命令始终确认
- 未加过滤条件的「删除所有节点」默认拒绝
- 单次写操作默认最多影响 50 个节点；超限时要求缩小范围
- 工具调用必须来自完整 `tool.call.final` 并通过 Tool Registry
- URL、路径、文件内容和网页内容一律视为不可信输入
- 文件读取必须绑定当前 `projectId`、`conversationId` 和 grant
- 禁止 `eval`、`Function()`、Shell 和动态模块执行

### 8.2 外部数据发送

首次启用 P0-A AI 助手、P1 工具或 P2 媒体能力时，界面需要说明以下内容：

- 将使用的模型和供应商
- 将发送的字段
- 是否包含 label、prompt、output 或媒体
- 远端服务可能产生的费用
- 用户如何关闭云端解析

`describe` 在发送媒体前必须获得本次操作确认。对于本地文件，需要说明媒体可能先上传到可被模型访问的远端地址。

联网搜索会把查询文本发送给搜索 Provider；读取本地文件只代表应用获得本地访问权，不自动代表可以把内容发送给云端模型。首次发送文件片段时，界面需要展示文件名、发送字符数、模型和 Provider，并允许仅本次或本会话授权。

### 8.3 提示注入防护

发送给 AI 的画布数据需要放入明确的数据字段，system message 必须声明其内容不可执行。即使节点 label 或 prompt 包含「忽略之前的规则」等文本，模型也只能输出允许的 `CommandIntent`，本地规划器仍会执行完整校验。

## 9. 并发、项目切换与异常恢复

聊天请求按会话隔离并受全局并发上限控制，但画布仍可能被鼠标、快捷键或其他面板修改。队列化不能替代项目、会话和画布版本校验。

| 场景 | 处理策略 |
|---|---|
| 同一会话连续发送消息 | 该会话内按顺序排队，上一条结束后处理下一条 |
| 不同会话同时发送消息 | 最多并发 2 个 AI 请求，其余请求排队 |
| 用户切换会话 | 原流继续写入原 `conversationId`，当前视图加载目标会话 |
| 用户切换项目 | 取消旧项目未执行计划，保留只读日志 |
| 确认期间画布变化 | 比较 `baseRevision`，重新规划并提示预览已更新 |
| 目标节点被删除 | 重新解析目标；目标为空时不执行 |
| 输入为空 | 忽略，不创建消息 |
| 输入超过 2,000 字符 | 拒绝解析并提示缩短，不截断后执行 |
| 画布为空 | 返回空画布提示，不执行写命令 |
| 首字节超过 10s | 取消 AI 请求，尝试本地规则降级 |
| 流连续 20s 无数据 | 标记响应中断，不执行未完成意图 |
| AI 连续失败 3 次 | 本次会话关闭 AI 解析并提示降级 |
| 搜索 Provider 失败 | 返回已知结果和错误，不自动切换付费 Provider |
| 网页被安全策略拒绝 | 显示拒绝原因，不把页面交给模型 |
| 文件 grant 已撤销 | 停止读取并请求重新授权 |
| 文件在授权后变化 | 重新校验真实路径、大小和类型 |
| 工具预算耗尽 | 停止工具循环并列出未完成步骤 |
| 应用关闭 | 取消未执行计划；已有 Store 变更走现有保存流程 |
| 历史已空 | 返回「没有可撤销的操作」 |

聊天面板关闭后，活动流和持久化继续运行。页面刷新或应用重启后，系统从 IndexedDB 恢复会话列表和最近消息；遗留的 `streaming` 消息恢复为 `interrupted`。操作日志必须携带 `projectId` 和 `conversationId`，避免跨项目或跨会话显示错误状态。

## 10. UI 与交互

对话面板从 Sidebar 入口打开，并作为 App 根布局的独立面板装配。现有可复用编辑器是 `MentionEditor`，不是 `MentionInput`；复用前需要确认其节点上下文依赖是否适合聊天输入。

### 10.1 消息状态

每条消息使用明确状态：

```typescript
type ChatMessageStatus =
  | 'queued'
  | 'parsing'
  | 'streaming'
  | 'clarifying'
  | 'preview'
  | 'executing'
  | 'done'
  | 'partial'
  | 'interrupted'
  | 'error'
  | 'canceled';
```

### 10.2 流式响应体验

助手收到 `start` 后展示模型和停止按钮；收到 `text.delta` 后增量更新同一条消息，不创建大量独立气泡。状态事件使用独立的轻量提示，不把模型的内部推理内容展示给用户。

用户停止或网络中断后，已接收文本保留在消息中，并显示“已停止”或“响应中断”。只有完成的 `intent.final` 才能显示命令预览；半截工具参数、JSON 或文本不得转成写操作。

### 10.3 指令预览

预览卡需要展示：

- 将执行的命令
- 目标节点及数量
- 参数变更前后值
- 将新增或删除的边、组和文件引用
- 是否发送数据给外部服务
- 是否可撤销
- 确认和取消按钮

低风险只读查询可以直接执行。删除、覆盖、批量修改和外部数据发送必须等待确认。

### 10.4 操作日志

每条日志记录以下信息：

- `projectId`、计划 ID 和命令 ID
- 解析来源：AI 或本地规则
- AI 请求 ID、模型 ID 和流结束原因
- 目标数量和执行结果
- 开始时间、耗时和错误码
- 对应历史索引
- 是否仍可撤销

日志不得记录 API Key。默认也不记录完整 prompt、output 和媒体 URL。

### 10.5 无障碍要求

- 输入、预览、确认和取消支持完整键盘操作
- 对话状态通过 `aria-live` 通知屏幕阅读器
- 流式 token 不逐字播报，每批次完成后合并播报
- 确认弹窗具备焦点陷阱，关闭后焦点返回触发元素
- `Escape` 只关闭当前浮层，不执行或丢弃已确认命令
- 尊重减少动画偏好
- 状态不能只依赖颜色区分

### 10.6 会话列表与历史入口

桌面宽屏采用“会话列表 + 当前对话”的双栏布局，小窗口把会话列表作为覆盖式抽屉。会话列表顶部提供新建和搜索，列表项显示标题、最近消息时间、流状态、未读、置顶和归档状态。

每个会话的上下文菜单包含重命名、置顶、归档、清空消息和移入回收站。回收站使用独立筛选视图，提供恢复、永久删除和清空回收站；破坏性操作不能只依赖图标表达。

### 10.7 工具调用与来源展示

工具调用在消息流中显示为可折叠卡片。卡片包含工具名称、输入摘要、授权状态、运行时间、停止按钮和结果摘要；涉及联网时显示 Provider 与域名，涉及文件时只显示文件名和授权范围。

联网回答在相关段落附近显示来源编号，消息底部提供来源列表。来源卡片展示标题、域名、发布时间或检索时间，并区分“搜索摘要”和“已读取正文”。文件引用展示文件显示名和行号，不显示绝对路径。

## 11. 多会话与历史记录管理

每个对话会话归属于一个画布项目。会话切换只改变 AI 上下文和聊天视图，不切换项目，也不修改画布状态。

### 11.1 三类历史的边界

产品中存在三类不同历史，UI 和数据结构不能混用：

| 历史类型 | 内容 | 删除影响 |
|---|---|---|
| 对话历史 | 用户消息、AI 流式回答和会话元数据 | 只影响聊天与后续 AI 上下文 |
| 操作日志 | 助手执行过的画布命令和结果 | 删除对话后仍保留最小审计信息 |
| 画布撤销栈 | nodes、edges 和 groups 快照 | 只通过 `undo()`、`redo()` 管理 |

删除对话不会撤销画布操作，也不会清理输出历史。清空画布撤销栈也不会删除聊天消息。

### 11.2 会话与消息模型

会话元数据用于列表、排序和回收站，不存放完整消息数组。

```typescript
interface ChatConversation {
  id: string;
  projectId: string;
  title: string;
  titleSource: 'auto' | 'user';
  pinned: boolean;
  archived: boolean;
  createdAt: number;
  updatedAt: number;
  lastMessageAt?: number;
  lastMessagePreview?: string;
  messageCount: number;
  deletedAt?: number;
}
```

每条消息单独持久化，并通过递增的 `sequence` 保持稳定顺序。

```typescript
interface PersistedChatMessage {
  id: string;
  projectId: string;
  conversationId: string;
  sequence: number;
  role: 'user' | 'assistant';
  content: string;
  status: ChatMessageStatus;
  requestId?: string;
  modelId?: string;
  createdAt: number;
  updatedAt: number;
  finishReason?: FinishReason;
  intents?: CommandIntent[];
}
```

### 11.3 会话操作

P0-A 支持以下会话管理操作：

| 操作 | 行为 | 是否确认 |
|---|---|---|
| 新建 | 在当前项目创建空会话并切换 | 否 |
| 切换 | 分页加载目标会话最近消息 | 否 |
| 自动命名 | 首轮完成后根据内容生成标题 | 否，可手动修改 |
| 重命名 | 更新标题并把 `titleSource` 设为 user | 否 |
| 置顶或取消置顶 | 调整会话列表排序 | 否 |
| 归档或取消归档 | 从默认列表隐藏或恢复 | 否 |
| 搜索 | P0 搜索标题和最近消息摘要 | 否 |
| 清空消息 | 保留会话元数据，删除全部聊天消息 | 是 |
| 从此处删除 | 删除选中消息及其后的消息 | 是 |
| 移入回收站 | 设置 `deletedAt`，保留可恢复数据 | 是 |
| 恢复 | 清除 `deletedAt` 并恢复到原项目 | 否 |
| 永久删除 | 删除会话和消息，保留脱敏操作日志 | 是，且不可撤销 |
| 清空回收站 | 永久删除回收站中的全部会话 | 是，且不可撤销 |

单独删除中间消息会破坏后续对话的因果上下文，因此不提供任意单条删除。用户可以选择“从此处删除”，系统删除该消息和所有后续消息，并取消关联的活动流与未确认计划。

### 11.4 会话切换与活动流

同一项目内切换会话时，原会话的流式请求可以在后台继续。会话列表显示生成中、已完成或失败徽标；切回后恢复当前增量文本。

并发规则如下：

- 每个会话最多有一个活动 AI 请求
- 应用默认最多同时运行 2 个会话请求
- 超过上限的新请求进入队列，用户可以取消排队
- 删除、清空或永久删除会话前先取消其活动流
- 切换画布项目时取消旧项目的活动流和未确认计划
- 后台完成的会话增加未读标记，但不弹出命令确认框
- 写命令预览只在对应会话处于前台时展示

### 11.5 AI 上下文隔离

每次 AI 请求只读取当前 `conversationId` 的消息。其他会话、归档会话和回收站会话不会进入上下文。

上下文构建遵循以下规则：

- 按 `sequence` 读取最近的已完成消息
- `streaming`、`interrupted` 和 `canceled` 消息默认不进入下一轮上下文
- 达到模型上下文预算时从最早消息开始裁剪
- P1 可以生成本地摘要，但摘要仍绑定原 `conversationId`
- 切换会话时取消旧的上下文构建任务，避免结果写入新会话
- 每个请求携带 `projectId`、`conversationId` 和 `requestId` 三重校验字段

### 11.6 IndexedDB 持久化

当前 `indexedDbService.ts` 使用 `ai-canvas-db`，数据库版本为 v8。实现多会话时需要提升版本并新增以下 Store：

| Store | keyPath | 索引 |
|---|---|---|
| `chatConversations` | `id` | `[projectId, updatedAt]`、`deletedAt`、`pinned` |
| `chatMessages` | `id` | `[conversationId, sequence]`、`requestId` |

升级过程只新增 Store 和索引，不改写现有项目、输出历史或配置记录。`onupgradeneeded` 失败时保持旧数据库可用，并关闭聊天持久化入口，不能删除或重建整个数据库。

持久化策略如下：

- 会话列表按 `updatedAt` 分页，每页 30 条
- 消息按 `sequence` 倒序分页，每页 50 条
- Zustand 只保留活动会话和最近访问会话的窗口数据
- 流式文本每 1s 或累计 2 KB 后写入临时检查点
- 收到 `done`、`error`、`canceled` 或窗口关闭事件时立即落盘
- 启动时把遗留的 `streaming` 消息恢复为 `interrupted`
- IndexedDB 配额不足时停止新增持久化，并提示导出或删除历史

### 11.7 自动标题与搜索

首轮对话完成后，助手可以根据首条用户消息和最终回答生成不超过 20 个汉字的标题。标题优先从同一次 AI 响应的元数据获得，不得为自动命名单独发起付费请求；标题缺失时使用首条用户消息的前 20 个可见字符。用户手动重命名后不得再次自动覆盖。

P0-A 只搜索标题和 `lastMessagePreview`，避免扫描所有消息。P1 如需全文搜索，应新增本地倒排索引或等价方案，不能在每次输入时调用 `getAll()` 遍历全部消息。

### 11.8 删除、恢复与数据边界

移入回收站属于软删除。会话和消息继续保存在本地，但不会出现在默认列表、AI 上下文或搜索结果中。

永久删除需要在一个 IndexedDB 事务中删除会话元数据和全部消息。删除完成后：

- 如果删除的是活动会话，切换到最近使用的可用会话
- 当前项目没有其他会话时创建一个新的空会话
- 保留操作日志中的命令 ID、时间和结果，不保留完整 prompt 或回答
- 不修改画布撤销栈、节点输出历史或本地媒体文件
- 明确提示本地删除不会删除 AI 供应商可能保留的请求日志

## 12. 联网搜索与本地文件工具

联网和文件能力通过受控 Tool Registry 提供。AI 只能提出结构化工具调用，不能直接发起 `fetch`、调用 Tauri FS、访问原始文件路径或执行系统命令。

### 12.1 工具注册表

每个工具需要独立输入 schema、风险级别、授权函数、执行函数和结果裁剪器。

```typescript
type ToolId =
  | 'web.search'
  | 'web.openResult'
  | 'file.requestGrant'
  | 'file.readText'
  | 'file.searchText'
  | 'file.list'
  | 'file.importToCanvas'
  | 'file.exportText';
```

注册项定义工具的安全边界和结果摘要方式：

```typescript
interface AgentToolDefinition<TInput, TOutput> {
  id: ToolId;
  category: 'web-read' | 'file-read' | 'canvas-write' | 'file-export';
  validateInput: (input: unknown) => TInput;
  authorize: (ctx: ToolContext, input: TInput) => Promise<ToolAuthorization>;
  execute: (ctx: ToolContext, input: TInput) => Promise<TOutput>;
  summarizeForModel: (output: TOutput) => ToolResultSummary;
}

interface ProposedToolCall {
  callId: string;
  toolId: ToolId;
  input: unknown;
}
```

`ToolAuthorization` 只有 `allowed`、`confirm` 和 `denied` 三种结果。模型不能自行构造 `allowed`，授权结果只由本地策略和用户操作产生。

工具调用流程如下：

```text
AI tool.call.final
  → Tool Registry 查找命令
  → 输入 schema 校验
  → 权限和风险策略
  → 必要时等待用户确认
  → 工具执行
  → 结果限额、脱敏和注入标记
  → ToolResult 返回同一会话的 AI
  → AI 继续流式回答或提出下一步工具
```

### 12.2 初始工具清单

P1 只提供读操作、画布导入和显式导出，不提供任意文件修改。

| 工具 ID | 能力 | 默认授权 | 阶段 |
|---|---|---|---|
| `web.search` | 查询搜索 Provider，返回标题、摘要和 URL | 每会话首次确认 | P1-A |
| `web.openResult` | 读取已验证的搜索结果正文 | 显示目标域名 | P1-A |
| `file.requestGrant` | 打开文件或目录选择器 | 始终由用户选择 | P1-B |
| `file.readText` | 读取已授权文本文件 | grant 范围内允许 | P1-B |
| `file.searchText` | 在已授权目录中搜索文本 | 目录 grant 内允许 | P1-B |
| `file.list` | 列出授权目录中的有限条目 | 目录 grant 内允许 | P1-B |
| `file.importToCanvas` | 复用 `fileService` 创建源节点 | 执行前预览 | P1-B |
| `file.exportText` | 通过保存对话框导出回答 | 每次选择保存路径 | P1-B |

以下工具不注册，也不接受模型用相近名称绕过：

- 删除、移动、重命名或覆盖任意本地文件
- 执行文件、Shell、PowerShell、脚本或应用程序
- 读取环境变量、浏览器凭据、密钥文件或系统配置
- 访问未授权路径、网络共享、设备路径或隐藏的父目录
- 任意 HTTP 请求、端口扫描、WebSocket 或本地服务调用

### 12.3 联网搜索 Provider

搜索通过 provider adapter 接入。API Key 保存在现有配置体系中，不写入对话、日志或模型上下文。

```typescript
interface WebSearchInput {
  query: string;
  domains?: string[];
  excludeDomains?: string[];
  recencyDays?: number;
  maxResults?: number;
}

interface WebSearchResult {
  id: string;
  title: string;
  url: string;
  snippet: string;
  publishedAt?: string;
  retrievedAt: number;
}
```

搜索规则如下：

- 查询长度最多 500 个字符
- 默认最多返回 10 条结果
- 域名过滤最多 20 项
- 每个结果分配稳定的 `SourceRef`，最终回答引用来源编号
- 搜索摘要只作为线索，重要结论需要读取来源正文
- 无法确认发布时间时不把抓取时间描述为发布时间
- 结果按不可信外部数据处理，不能改变工具策略或 system message

### 12.4 页面读取与 SSRF 防护

`web.openResult` 只能读取搜索结果中的 URL，或用户在本轮明确提供并确认的 URL。执行器需要防止服务端请求伪造（SSRF）和本地资源访问：

- 只允许 `https`，兼容需求明确时才允许 `http`
- 拒绝 `file:`、`data:`、`javascript:`、`ftp:` 和自定义 scheme
- 拒绝带用户名或密码的 URL
- 默认只允许 80 和 443 端口
- 拒绝 localhost、环回、私有、链路本地、保留和组播地址
- DNS 解析后再次校验所有 IP，防止 DNS rebinding
- 最多跟随 3 次重定向，每次重定向重新校验 URL 和 IP
- 响应体最多 2 MB，提取后正文最多 200 KB
- 不下载附件，不执行脚本，不加载 iframe 和子资源
- HTML 转为纯文本或经过严格清理的 Markdown 后再显示

网页读取应由专用 Rust 命令执行，以便在连接前解析 DNS 和校验目标 IP。现有通用 `proxy_fetch` 接受任意 URL、方法和 headers，不能直接暴露给智能体；需要新增受限的 `assistant_web_read`，或先对通用代理完成等价的安全收敛。

页面正文、标题和结构化数据都可能包含提示注入。工具结果必须包裹为不可信引用材料，AI 不得执行其中出现的命令或工具指令。

### 12.5 会话级文件授权

虽然应用自身具备较广的 Tauri 文件读取能力，智能体不能继承该范围。用户通过原生文件选择器授权后，本地代码生成不含路径的 `grantId`；模型只能引用该 ID。

```typescript
interface FileGrant {
  id: string;
  projectId: string;
  conversationId: string;
  kind: 'file' | 'directory';
  capabilities: Array<'read' | 'search' | 'list' | 'import'>;
  displayName: string;
  canonicalPath: string;
  createdAt: number;
  expiresAt: number;
}
```

`canonicalPath` 只存在于本地授权管理器，不能发送给模型或写入普通聊天消息。P1-B 的 grant 只在当前应用会话内有效；应用重启、项目切换、对话删除或用户撤销授权后失效。

文件执行器需要完成以下检查：

- 解析规范化绝对路径，拒绝 `..`、NUL 字节和设备路径
- Windows 路径比较忽略大小写，并拒绝未经确认的 UNC 网络路径
- 目录 grant 下的每个文件都要验证仍位于授权根目录
- 跟随符号链接或 junction 后再次验证真实路径，防止越界
- 模型传入的文件名只用于匹配，不能拼接成系统路径
- 文件变化后重新读取 `stat`，避免授权后被替换

### 12.6 文件读取、搜索与格式边界

P1-B 首批支持 UTF-8 文本及可安全转为文本的格式：`.txt`、`.md`、`.json`、`.csv`、`.tsv`、`.log` 和项目源代码文本。扩展名和 MIME type 只能作为初筛，还需要检查文件大小、二进制特征和解码结果。

默认限制如下：

- 单文件最多读取 2 MB
- 单次发送给模型的文本最多 100,000 个字符
- 目录搜索最多扫描 200 个文件或累计 20 MB
- 搜索结果最多返回 100 个命中片段
- 每个片段携带 grant ID、显示文件名和行号，不携带绝对路径
- 超出上下文预算时按相关度裁剪，并告知用户未读取完整内容

PDF、Word、Excel 和其他复合文档不能当作纯文本读取。支持这些格式前，需要分别接入受控解析器、大小限制和解析失败处理；二进制媒体优先通过现有 `fileService` 导入画布，不直接把原始字节发送给文本模型。

### 12.7 导入画布与导出文件

`file.importToCanvas` 复用 `fileService` 既有上传和项目落盘能力，再通过 Store Action 创建源节点。工具层不能直接调用 `@tauri-apps/plugin-fs`，也不能绕开节点历史和文件清理逻辑。

`file.exportText` 每次都打开原生保存对话框。用户选择目标后才能写入；工具不能静默覆盖现有文件，也不能把模型提供的路径直接交给文件系统。

### 12.8 工具授权与用户反馈

工具卡片需要展示工具名称、发送或读取的数据范围、Provider、目标域名或文件显示名。授权策略如下：

| 操作 | 授权策略 |
|---|---|
| 首次联网搜索 | 显示搜索 Provider 和查询文本，可允许本会话 |
| 后续同会话搜索 | 使用会话授权，仍展示可停止的工具卡片 |
| 打开网页 | 展示域名和来源标题 |
| 选择文件或目录 | 必须由用户操作原生选择器 |
| 读取已授权文件 | 显示文件名、大小和读取范围 |
| 导入画布 | 展示将创建的节点并确认 |
| 导出文本 | 每次使用保存对话框确认路径 |

用户可以在会话工具面板撤销联网许可或文件 grant。撤销后，排队中的相关工具调用失败，新的 AI 请求不能继续引用该 grant。

### 12.9 工具循环、预算与中止

一次用户消息可以触发多轮“AI → 工具 → AI”，但需要限制成本和递归：

- 每条消息最多 5 轮模型调用
- 最多执行 8 个工具调用
- 最多打开 5 个网页正文
- 最多并行 3 个只读工具
- 工具累计运行时间最多 60s
- 工具结果发送给模型的总文本最多 200 KB
- 用户停止时同时中止模型流和活动工具
- 工具不能直接调用另一个工具，只能把结果返回 AI

达到预算后，助手停止继续调用工具，列出已完成结果和未完成步骤，并让用户决定是否继续新一轮。

### 12.10 历史记录与工具数据

对话历史持久化工具 ID、调用状态、来源引用和脱敏结果摘要。默认不持久化完整网页正文、本地文件内容、绝对路径或 API Key。

删除会话时需要同时清理该会话缓存的网页正文、来源卡片和失效 grant 元数据。删除聊天不会删除原始本地文件，也不会请求搜索 Provider 删除其服务端日志。

## 13. 多媒体生成与双交付模式

多媒体生成分为两个独立动作：`media.generate*` 调用模型并产生 Artifact，`canvas.materializeArtifact` 把已有 Artifact 物化为节点。聊天呈现和画布节点可以独立成功或失败。

### 13.1 支持的媒体工具

文本助手通过 Tool Registry 调用下列能力：

| 工具 ID | 产物 | 典型输入 |
|---|---|---|
| `media.generateImage` | 图片 | prompt、参考图、比例、画质和风格 |
| `media.generateVideo` | 视频 | prompt、首尾帧、时长、比例、分辨率和音频开关 |
| `media.generateMusic` | 音乐 | prompt、歌词、时长、风格和纯音乐开关 |
| `media.synthesizeSpeech` | 语音 | text、voice、语言、语速、情感和音色参考 |
| `media.getTaskStatus` | 任务状态 | task ID |
| `media.cancelTask` | 取消结果 | task ID |
| `canvas.materializeArtifact` | 画布节点 | artifact ID、位置和连接关系 |

音乐与语音不能只依赖 `audio` category 区分。模型需要分别声明 `text-to-music`、`text-to-speech`、`voice-clone` 等 capability，并使用不同参数 schema。

### 13.2 三种交付模式

每次媒体生成都携带 `deliveryMode`：

| 模式 | 对话表现 | 画布表现 | 典型表达 |
|---|---|---|---|
| `chat` | 完整媒体结果卡 | 不创建节点 | 「生成一张图给我看看」 |
| `canvas` | 显示进度和节点入口 | 创建并更新节点 | 「在画布生成一张图」 |
| `both` | 完整媒体结果卡 | 节点引用同一产物 | 「生成图片并放到画布」 |

意图不明确时默认使用 `chat`。聊天结果卡提供「添加到画布」按钮；点击后只调用 `canvas.materializeArtifact`，不能再次调用生成模型。

### 13.3 生成请求、任务与产物

生成请求描述用户意图和交付目标：

```typescript
type MediaKind = 'image' | 'video' | 'music' | 'speech';
type DeliveryMode = 'chat' | 'canvas' | 'both';

interface MediaGenerationRequest {
  projectId: string;
  conversationId: string;
  messageId: string;
  kind: MediaKind;
  prompt: string;
  modelRef?: string;
  deliveryMode: DeliveryMode;
  inputRefs: string[];
  params: Record<string, unknown>;
}
```

任务统一承载供应商提交、轮询、取消和结果：

```typescript
interface GenerationTask {
  id: string;
  request: MediaGenerationRequest;
  status: GenerationTaskStatus;
  providerTaskId?: string;
  artifactId?: string;
  targetNodeId?: string;
  errorCode?: string;
  createdAt: number;
  updatedAt: number;
}
```

Artifact 是模型调用的唯一产物记录，聊天消息和节点只保存引用：

```typescript
interface MediaArtifact {
  id: string;
  projectId: string;
  kind: MediaKind;
  filePath?: string;
  remoteUrl?: string;
  mimeType: string;
  modelId: string;
  provider: string;
  prompt: string;
  metadata: Record<string, unknown>;
  createdAt: number;
}
```

### 13.4 独立状态机

模型生成、聊天呈现和节点物化使用三组状态，不能共用一个 `message.status`：

```typescript
type GenerationTaskStatus =
  | 'awaiting-confirmation'
  | 'queued'
  | 'submitting'
  | 'polling'
  | 'succeeded'
  | 'failed'
  | 'canceled';

type ChatPresentationStatus =
  | 'hidden' | 'pending' | 'visible' | 'failed';

type CanvasMaterializationStatus =
  | 'none' | 'pending' | 'created' | 'failed';
```

`both` 模式下，聊天呈现失败不应删除已经成功的节点；节点创建失败也不应隐藏已经生成的聊天产物。UI 分别显示两个结果和重试入口。

### 13.5 交付流程

聊天模式不提前创建节点：

```text
media.generate*
  → 创建聊天结果占位卡
  → Generation Runtime 执行
  → Artifact 保存到项目数据目录
  → 消息引用 artifactId 并展示媒体
```

画布模式先创建可追踪的占位节点：

```text
media.generate*
  → 创建 loading 节点并记录 targetNodeId
  → Generation Runtime 执行
  → Artifact 保存
  → 节点回填 artifactId 和本地 URL
  → 聊天显示节点入口和任务结果
```

双端模式只调用一次模型：

```text
一个 GenerationTask
  → 一个 MediaArtifact
  → ChatMessage.artifactId
  → CanvasNode.data.artifactId
```

### 13.6 模型能力注册与默认值

`GeneralModelCategory` 只适合 UI 分组，不能承担路由。每个模型需要声明 capability、输入类型、参数 schema、进度和取消能力。

```typescript
interface MediaModelCapabilities {
  tasks: Array<
    | 'text-to-image'
    | 'image-to-image'
    | 'text-to-video'
    | 'image-to-video'
    | 'text-to-music'
    | 'text-to-speech'
    | 'voice-clone'
  >;
  supportsProgress: boolean;
  supportsCancel: boolean;
  parameterSchemaId: string;
}
```

默认模型配置应收敛为 capability 映射，避免继续增加平铺字段：

```typescript
interface AssistantModelDefaults {
  chat?: string;
  image?: string;
  video?: string;
  music?: string;
  speech?: string;
}
```

模型解析优先级如下：

1. 本轮 `@model` 显式覆盖
2. 当前会话的模型偏好
3. 应用的 capability 默认模型
4. 唯一可兼容模型
5. 无法确定时在工具预览卡中请求选择

### 13.7 `@` 引用与工具触发

自然语言和 `/image`、`/video`、`/music`、`/speech` 用于选择工具；`@` 用于引用输入实体或显式覆盖模型。用户不需要每次都 `@` 默认模型。

输入栏不应长期并列展示文本、图片和视频三个模型选择器。保留一个“对话助手模型”主选择器；图片、视频、音乐和语音模型放入默认能力设置，并通过 `@model` 快速覆盖。媒体工具预览卡在执行前再次展示实际模型并允许修改，这样既减少输入栏噪声，也不会隐藏最终调用对象。

建议使用类型化标记：

```text
@node{node_id|label}
@asset{asset_id|label}
@model{model_id|label}
@voice{voice_id|label}
```

示例：

- 「用 `@model{seedance_2|Seedance 2}` 把 `@node{node_12|首帧}` 做成 5 秒视频」
- 「用 `@model{suno_v4|Suno V4}` 生成纯音乐，在对话里播放」
- 「用 `@voice{voice_7|女声 7}` 朗读 `@node{node_3|旁白}` 并放到画布」

当前 `MentionEditor` 的节点标记 `@{nodeId:label}` 需要保持兼容。模型、声音和资产使用独立前缀，不能把模型 ID 塞入节点标记。

### 13.8 工具预览与确认

媒体生成属于可能付费的外部副作用。执行前的工具卡需要展示：

- 媒体类型和交付模式
- Provider、模型和能力
- prompt、参考节点和素材
- 关键参数和预计产物数量
- 已知价格或费用不可预估提示
- 将创建的节点、连接和画布位置
- 取消和确认按钮

用户可以开启“本会话内使用该模型免确认”，但需要设置单次和累计预算上限。批量生成、模型切换、声音克隆和未知价格仍需确认。

### 13.9 Generation Runtime

ChatPanel 不能直接调用 `generateImage`、`generateVideo` 或 `generateAudio`。统一 Generation Runtime 负责：

- 根据 capability 解析模型和 adapter
- 校验并映射 provider 参数
- 创建任务、提交请求和恢复轮询
- 归一化进度、错误、取消和结果
- 将远端结果保存到项目数据目录
- 创建 Artifact 并回填消息或节点
- 记录模型、参数、费用和输出历史

现有 provider 和 workflow 分流继续复用。Runtime 不应新增平行的 `if (model === ...)` 分支。

### 13.10 产物持久化与生命周期

聊天模式产生的媒体也要保存为项目 Artifact，不能只保留可能过期的远端 URL。会话历史保存 `artifactId`，媒体文件继续由项目数据目录管理。

删除会话时：

- 被画布节点或其他会话引用的 Artifact 保留
- 只被该会话引用的 Artifact 移入可恢复回收区
- 永久删除前再次检查引用计数
- 删除聊天不能误删节点正在使用的本地文件

将聊天产物添加到画布时增加引用，不复制文件，也不创建第二条生成历史。

### 13.11 流式进度与取消

模型流和媒体生成任务使用不同事件。媒体 Runtime 向聊天和节点发布统一事件：

```typescript
type MediaTaskEvent =
  | { type: 'media.queued'; taskId: string }
  | { type: 'media.submitted'; taskId: string }
  | { type: 'media.progress'; taskId: string; progress?: number }
  | { type: 'media.artifact'; taskId: string; artifactId: string }
  | { type: 'media.error'; taskId: string; code: string }
  | { type: 'media.done'; taskId: string };
```

取消时需要区分远端取消和停止跟踪。聊天卡与节点必须显示相同任务状态，但各自可以独立隐藏或移除。

### 13.12 迁移当前媒体调用方式

现有通过 `generate_image`、`generate_video` 代码块和正则识别的方式只作为迁移对象。它存在以下限制：

- 流式 JSON 可能被截断
- 媒体类型容易在回调中丢失
- 配置图片模型后可能把普通聊天误判为生图
- ChatPanel 直接依赖生成服务
- 无法扩展音乐、语音、引用输入和交付模式

迁移后只接受通过 schema 校验的 `tool.call.final`。旧代码块解析器在工具协议稳定后删除，不能与新协议长期并存。

## 14. 非功能需求

这些指标用于限制首版复杂度，并为性能和稳定性测试提供基线。

| 类别 | 阶段目标 |
|---|---|
| 本地解析性能 | 500 个节点以内，规则解析和规划目标为 100 ms p95 |
| 交互反馈 | 提交输入后 100 ms 内显示 queued 或 parsing 状态 |
| 首个流事件 | 可控网络下，3s p95 内收到 `start` 或首个 `text.delta` |
| UI 刷新节流 | `text.delta` 最多每 50 ms 合并写入 Store 一次 |
| 流超时 | 首字节 10s、空闲 20s、总时长 120s |
| 流大小上限 | 单次响应最多接收 1 MB，超限后中止 |
| 批量节点上限 | 默认 50 个，超限时拒绝执行 |
| 批量连线上限 | 默认新增 100 条边 |
| 会话切换 | 缓存会话 100 ms 内展示；未缓存会话 300 ms p95 内展示首屏 |
| 会话列表分页 | 每页 30 条，不一次性加载全部会话 |
| 消息窗口 | 每个已加载会话最多保留最近 100 条，旧消息按 50 条分页 |
| 流式持久化 | 每 1s 或累计 2 KB 写检查点，结束状态立即落盘 |
| AI 并发 | 每个会话 1 个活动请求，应用默认最多 2 个 |
| 工具循环 | 每条消息最多 5 轮模型调用和 8 个工具调用 |
| 联网结果 | 搜索最多 10 条，正文最多 5 页，每页提取不超过 200 KB |
| 联网安全 | 所有 URL、重定向和解析后 IP 均通过 SSRF 校验 |
| 文件读取 | 单文件 2 MB，单轮发送模型最多 100,000 字符 |
| 目录搜索 | 最多 200 个文件、20 MB 和 100 个命中片段 |
| 工具总时长 | 每条消息累计最多 60s，用户停止后立即取消 |
| 媒体任务并发 | 应用默认最多 2 个活动任务，并允许按 Provider 能力收紧 |
| 双端交付 | `both` 只能创建一个任务和一个 Artifact，禁止重复调用或重复计费 |
| 产物完成条件 | 远端结果成功保存并创建 Artifact 后，任务才能标记为 `succeeded` |
| 状态一致性 | 任务状态唯一；聊天呈现和节点物化分别记录，不互相覆盖 |
| 费用确认 | 执行前展示模型、数量和可获得的价格信息；未知价格必须明确提示 |
| 一致性 | 每个原子画布命令只产生一个历史步骤 |
| 数据最小化 | P0-A 默认不发送 label、prompt、output、路径和媒体 |
| 可诊断性 | 每次执行返回稳定错误码和计划 ID |
| 成本控制 | 每条消息最多 5 轮 AI 请求，不自动重试，超限需用户继续 |

性能目标需要在项目约定的开发基准机上测量。若基准结果不满足目标，应先优化 Selector 解析和快照构建，不能通过跳过安全校验换取速度。

## 15. 关键架构决策

以下决策作为本方案内嵌的架构决策记录。进入实施后，如决策影响多个平台能力，应迁移为独立 ADR。

### ADR-001：AI 主导对话，本地代码控制执行

**状态**：接受

**背景**：对话助手需要 AI 提供自然语言理解和连续对话体验，但模型输出不稳定，不能承担目标解析、权限和确认职责。

**决策**：P0-A 默认使用 AI 生成流式回答和结构化意图。模型只输出命令 ID、Selector AST、候选参数和置信度；节点 ID、参数白名单、风险和确认策略全部由本地代码计算。本地规则仅承担核心命令降级。

**代价**：P0-A 必须同时建设模型配置、流式 Adapter、本地 Selector 和 command schema，但产品从首版就具备 AI 对话体验，安全边界仍由本地代码控制。

### ADR-002：批量写操作使用专用原子 Action

**状态**：接受

**背景**：现有单节点 Action 会各自提交历史，循环调用无法一次撤销。

**决策**：为批量删除、连接和分组提供专用原子 Action，不在助手层循环拼接单节点 Action。

**代价**：Store 需要新增批量能力，但不会引入全局“跳过历史”开关。

### ADR-003：重生成延后到统一生成运行时之后

**状态**：接受

**背景**：当前生成流程分布在多个节点组件中，助手无法可靠复用。

**决策**：P0 不实现重生成。先抽出统一生成入口，再让 UI 和助手共同调用。

**代价**：用户稍后才能使用对话重生成，但避免复制 provider 和 workflow 分支。

### ADR-004：取消区分远端取消与停止跟踪

**状态**：接受

**背景**：`cancelNodePolling` 不能保证远端任务停止。

**决策**：每个 provider 声明取消能力，UI 根据实际结果展示不同文案。

**代价**：需要扩展 provider 能力描述，但不会向用户承诺不存在的取消能力。

### ADR-005：统一模型流为领域事件

**状态**：接受

**背景**：不同供应商使用 SSE、NDJSON、流式 tool call 或自定义分片。UI 直接依赖供应商格式会导致解析、取消和错误处理散落。

**决策**：每个模型 Adapter 把供应商响应转换为 `AssistantStreamEvent`。UI 和规划器只消费统一事件；只有完整的 `intent.final` 可以进入本地执行管线。

**代价**：需要维护流解析器和 Adapter 契约，但可以统一处理 UTF-8 分片、背压、中止、usage 和断流恢复。

### ADR-006：会话按项目隔离并独立持久化

**状态**：接受

**背景**：把所有消息放入单个 Chat Slice 或 `CanvasProject` 会导致上下文串用、项目记录膨胀和切换时全量加载。

**决策**：会话必须包含 `projectId`，消息必须包含 `conversationId`。会话元数据和消息分别写入 IndexedDB Store，Zustand 只缓存活动窗口。

**代价**：需要数据库版本升级、分页服务和缓存淘汰逻辑，但会话切换、恢复和删除具有稳定边界。

### ADR-007：删除会话先进入回收站

**状态**：接受

**背景**：对话历史可能包含重要提示词和操作依据，直接硬删除难以恢复。

**决策**：普通删除只设置 `deletedAt`。永久删除和清空回收站需要再次确认，并在一个事务中删除会话及其消息。

**代价**：回收站数据继续占用本地空间，但能降低误删风险，并使恢复行为可预测。

### ADR-008：模型只使用注册工具和授权句柄

**状态**：接受

**背景**：应用级网络和文件权限大于单次对话实际需要。把 URL 或路径直接交给模型会扩大 SSRF、路径穿越和数据外泄风险。

**决策**：AI 只能提出注册工具调用。本地文件使用会话级 `grantId`，联网使用受限 Provider 和安全网页读取器；模型不能访问原始路径或通用网络代理。

**代价**：用户需要进行显式授权，工具实现需要额外的 schema 和策略层，但权限可以审计、撤销和按会话隔离。

### ADR-009：网页与文件内容始终视为不可信数据

**状态**：接受

**背景**：搜索结果、网页和本地文档可能包含提示注入、恶意链接或伪造指令。

**决策**：工具结果经过大小限制、内容清理和不可信标记后才能进入模型上下文。工具结果不能修改 system message、授权状态或工具预算。

**代价**：模型可能无法完整读取超大内容，但能防止外部材料直接控制智能体。

### ADR-010：生成任务与节点物化分离

**状态**：接受

**背景**：媒体既可以只在对话中呈现，也可以生成画布节点。如果把模型调用绑定到节点创建，聊天模式无法独立工作，双端模式还可能重复调用和重复计费。

**决策**：`media.generate*` 只负责创建一个 `GenerationTask` 和一个 `MediaArtifact`；聊天消息和画布节点分别引用该 Artifact。`canvas.materializeArtifact` 只把已有产物物化为节点，不能重新调用模型。

**代价**：需要新增任务、产物和引用生命周期，但可以独立重试聊天呈现或节点物化，并保证双端结果一致。

### ADR-011：`@` 用于显式引用，不作为媒体能力的唯一入口

**状态**：接受

**背景**：如果要求用户每次通过 `@模型` 触发生图、生视频、音乐或语音，输入成本高，也会把工具选择与模型选择耦合。

**决策**：助手根据自然语言或斜杠命令选择媒体工具；`@model`、`@node`、`@asset` 和 `@voice` 用于显式覆盖模型或引用输入。没有 `@model` 时按会话偏好和 capability 默认值路由。

**代价**：需要可靠的工具意图和 capability 校验，但普通用户可以直接表达目标，高级用户仍能精确控制。

## 17. 测试策略

测试优先验证安全边界、目标解析和原子历史，不依赖真实付费模型。

### 17.1 规则与 Selector 单元测试

| 用例 | 预期 |
|---|---|
| 「删除失败的图片节点」 | `and(status=error, type=ai-image)` |
| 「删除图片和视频节点」 | `or(type=ai-image, type=ai-video)` |
| 「删除失败节点，不包括文本节点」 | `and(status=error, not(type=ai-text))` |
| 「选中 3 号节点」 | 匹配 `data.displayId === 3` |
| 「移除那个」且单选 | 解析为选中节点 |
| 「移除那个」且多选 | 反问，不执行 |
| 「再来一次」 | 不按多意图拆分 |
| 空输入 | 忽略 |
| 超过 2,000 字符 | 拒绝，不截断执行 |

### 17.2 命令与事务测试

| 用例 | 验证点 |
|---|---|
| 删除 3 个节点 | Store 一次更新，历史只推进一步 |
| 批量校验失败 | Store、历史和文件操作均不变化 |
| 删除包含子节点的组 | 节点、边和组引用一致 |
| 确认期间 revision 改变 | 原计划拒绝执行并重新预览 |
| 项目切换后执行旧计划 | 因 `projectId` 不匹配而拒绝 |
| 重复连线 | 不创建重复边 |
| 连线超过 100 条 | 拒绝并提示缩小范围 |
| 撤销最新助手命令 | 恢复完整批次 |

### 17.3 AI 契约与流式解析测试

P0-A 使用 fake stream adapter 返回固定事件，不要求真实 `generalModels`：

- 非法命令 ID 被拒绝
- AI 返回节点 ID 字段时被拒绝
- AI 返回 `requiresConfirm=false` 时被忽略或拒绝
- 未知参数字段被命令 schema 拒绝
- SSE 数据跨任意 chunk 边界仍能正确解析
- UTF-8 中文字符跨 chunk 时不会乱码
- 多个事件位于同一 chunk 时按顺序输出
- `conversation.title` 只更新自动标题，不覆盖用户标题
- `intent.final` 缺失或不完整时不生成执行计划
- `error` 或断流后保留文本但清除未完成意图
- 首字节、空闲和总时长超时都会触发 `AbortSignal`
- 用户停止后不再提交新的 `text.delta`
- 流超过 1 MB 时返回 `STREAM_LIMIT_EXCEEDED`
- 标签中的提示注入不改变本地策略
- 连续失败 3 次后切换到本地规则降级

真实模型只用于人工冒烟测试，并明确记录模型、供应商、事件顺序和费用。

### 17.4 多会话与持久化测试

- 同一项目创建多个会话后可独立切换和恢复消息
- 不同会话的 AI 请求只包含各自消息
- 会话切换期间完成的后台流写回原 `conversationId`
- 项目切换会取消旧项目活动流和未确认计划
- 应用重启后会话顺序、置顶、归档和未读状态保持一致
- 遗留 `streaming` 消息在启动后恢复为 `interrupted`
- 消息按 50 条分页且顺序稳定，不重复、不漏页
- 清空消息保留会话，移入回收站后可以恢复
- 永久删除同时清理会话和消息，不影响画布撤销栈
- 删除活动会话后自动选择可用会话或创建空会话
- IndexedDB 升级失败时不删除现有项目和历史 Store
- 配额不足时停止写入并向用户显示处理入口

### 17.5 联网与文件工具安全测试

- 未注册工具 ID 和未知参数字段被拒绝
- `tool.call.delta` 不触发工具，只有完整 `tool.call.final` 可进入注册表
- localhost、私有 IP、链路本地、保留地址和非 HTTP scheme 被拒绝
- DNS 解析到私有 IP 或重定向到私有 IP 时被拒绝
- 超过 3 次重定向、2 MB 响应或 200 KB 正文时中止
- 网页中的工具指令和提示注入不能改变 system message
- 模型提交绝对路径、`..`、NUL、UNC 或设备路径时被拒绝
- 文件 grant 不能跨项目、跨会话或重启继续使用
- 符号链接和 junction 解析后越过授权目录时被拒绝
- 文件变化后重新执行 `stat` 并校验授权范围
- 超过文件、目录、字符、工具轮数和执行时间上限时安全停止
- 撤销 grant 后，排队和后续文件工具调用失败
- 导出必须打开保存对话框，不能静默覆盖文件
- 删除会话会清理网页缓存和 grant，不删除原始本地文件
- 工具日志和错误信息不包含 API Key 或绝对路径

### 17.6 多媒体生成与双交付测试

- 普通问答不会因为配置了图片或视频模型而触发媒体工具
- `chat` 模式生成成功后展示媒体卡，且不创建节点
- `canvas` 模式创建占位节点并回填产物，聊天只显示进度和节点入口
- `both` 模式只提交一次 Provider 请求，并让消息和节点引用同一 `artifactId`
- 对话结果的「添加到画布」只物化已有 Artifact，不再次生成
- 聊天呈现失败不删除成功节点，节点物化失败不隐藏成功聊天产物
- `kind`、`deliveryMode` 和显式 `@model` 在流式工具参数中不会丢失
- 显式模型、会话偏好、应用默认值和唯一兼容模型按既定优先级解析
- capability 不兼容的模型在提交前被拒绝，不能自动换成其他媒体类型
- 音乐、语音和声音克隆使用不同 capability 与参数 schema
- 删除会话不会删除仍被节点引用的 Artifact；最后一个引用删除后才进入回收区
- Provider 不支持远端取消时只显示“停止跟踪”，不能显示“已取消生成”
- `tool.call.delta` 和不完整媒体参数不会创建任务或占位节点

### 17.7 UI 与无障碍测试

- 键盘完成输入、预览、确认和取消
- 焦点在弹窗关闭后正确返回
- 流式文本在同一消息中增量更新
- 高频 `text.delta` 按 50 ms 批量提交，不造成逐 token 重渲染
- 停止按钮中止请求并保留已接收文本
- `aria-live` 合并播报流式内容，不逐 token 播报
- 同一会话消息保持顺序，不同会话遵守全局并发上限
- 项目切换后不显示错误预览
- 取消确认没有 Store 或外部副作用
- reduced motion 环境关闭非必要动画

### 17.8 测试工具决策

当前项目未安装 Vitest 或其他单元测试框架。引入 Vitest 属于新增依赖，需要在实施前说明原因、替代方案和影响并等待确认。未获确认前，可以先完成纯函数设计和手动验收清单，但不能声称自动化测试通过。

## 20. 风险、回滚与开放问题

本节记录实施前需要控制的风险、阶段回滚边界和仍待确认的产品或技术选择。

### 20.1 主要风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| 批量 Action 破坏现有历史 | 撤销错误或状态丢失 | 先为事务和历史编写测试，保留单节点 Action |
| Selector 误判 | 操作错误节点 | 写操作预览、确认和 revision 校验 |
| 面板挤压画布 | 小窗口体验下降 | 独立布局测试，设置最小宽度和折叠模式 |
| AI 解析不稳定 | 频繁反问或错误意图 | AI 只产出意图，本地决定目标和策略 |
| 供应商流格式差异 | 分片丢失或事件乱序 | 每个 provider 使用 Adapter 并运行流契约测试 |
| 高频 token 更新 | React 重渲染和输入卡顿 | 50 ms 缓冲合并、消息级订阅和大小上限 |
| 半截 tool call 被误用 | 执行错误命令 | 仅接受完整 `intent.final`，断流时清除意图 |
| 会话上下文串用 | AI 获得错误项目或会话内容 | 请求携带三重 ID 校验并按会话读取消息 |
| IndexedDB 升级失败 | 聊天历史不可用 | 只新增 Store，失败时关闭持久化且保留旧数据 |
| 流式频繁落盘 | 写放大和 UI 卡顿 | 1s 或 2 KB 检查点，结束时强制刷新 |
| 误删重要会话 | 提示词和回答丢失 | 回收站、恢复和永久删除二次确认 |
| SSRF 或内网探测 | 访问本机、内网或云元数据 | 专用 Rust reader、DNS/IP 与重定向逐跳校验 |
| 网页提示注入 | 外部内容诱导额外工具或数据外泄 | 不可信标记、本地策略和工具预算不可被结果修改 |
| 文件授权越界 | 读取未选择文件或敏感目录 | grant ID、真实路径校验和符号链接越界检查 |
| 工具结果泄露路径 | 绝对路径进入模型或日志 | 只返回显示名和 grant ID，错误信息脱敏 |
| 工具循环失控 | API 费用和等待时间增加 | 轮数、调用数、并发、时长和结果大小预算 |
| 搜索来源不可靠 | AI 基于摘要输出错误结论 | 来源正文核验、引用卡片和检索时间标记 |
| 媒体外发不透明 | 隐私和费用风险 | 操作级许可和数据范围披露 |
| 供应商不支持取消 | 远端任务和费用继续 | capability 矩阵和准确文案 |
| 双端重复生成 | 重复计费且聊天与节点结果不一致 | 一个任务、一个 Artifact、两处引用，Provider 提交使用幂等键 |
| 交付状态互相覆盖 | 一端失败导致另一端成功结果消失 | 任务、聊天呈现和节点物化使用独立状态 |
| Artifact 成为孤儿 | 项目空间持续增长或误删共享文件 | 引用计数、回收区、永久删除前重查引用 |
| 模型路由错误 | 调错媒体模型或参数不兼容 | capability 注册、schema 校验和执行前预览 |
| 音乐与语音混用 | 错误模型和错误参数 | 使用独立 task capability，不按 `audio` 分类直接路由 |
| 远端 URL 过期 | 历史对话中的媒体无法访问 | 成功后保存到项目数据目录，再标记任务完成 |

### 20.2 回滚方案

每个阶段通过功能开关控制入口。回滚 P0-A 时关闭画布助手入口并停止创建新消息，不删除用户项目数据或已经写入的聊天 Store。IndexedDB 新增 Store 属于前向兼容变更，回滚代码可以停止读取，但不能降低版本并重建数据库。原子批量 Action 保留为 Store 能力时不得改变现有单节点 Action 行为。

回滚 P1-A 或 P1-B 时关闭对应 Tool Registry 条目，取消活动工具并撤销内存 grant。网页缓存和工具结果摘要可以清理，但不能删除用户原始文件。回滚不需要扩大或放松 Tauri capability，也不能退回直接暴露 `proxy_fetch` 或原始文件路径的实现。

回滚 P2-A 或 P2-B 时关闭媒体 Tool Registry 和节点物化入口，停止创建新任务。已有任务和 Artifact 继续可读，已完成媒体不能因功能回滚被删除；被聊天或节点引用的文件按原引用关系保留。不能退回 ChatPanel 直接调用 Provider 或通过代码块正则触发生成的方式。

### 20.3 实施前开放问题

- P0-A 的默认批量上限是否采用 50 个节点
- ChatPanel 使用固定宽度、可拖拽宽度还是覆盖式浮层
- `MentionEditor` 是否适合直接复用，或需要提取轻量输入能力
- Store revision 是全局画布计数还是每项目计数
- Vitest 与参数 schema 库是否允许作为新依赖
- 首批支持哪些供应商的 SSE 或 NDJSON 格式
- 非流式模型是否允许作为显式兼容选项
- 首字节 10s、空闲 20s、总时长 120s 是否作为默认超时
- 单次 1 MB 流大小上限是否需要按模型配置
- 首次选择助手模型后，云端解析是否立即启用或再次确认
- 回收站是否只手动清理，或增加可配置的自动清理周期
- 默认最多同时运行 2 个会话请求是否合适
- P0 搜索范围是否包含 `lastMessagePreview`
- 永久删除后保留多久的脱敏操作日志
- 首个搜索 Provider 采用哪种 API 和计费模型
- `web.openResult` 是否只允许 HTTPS，还是允许用户确认后的 HTTP
- 搜索结果正文缓存保留到会话结束还是应用关闭
- P1-B 首批文本扩展名是否包含项目源代码文件
- FileGrant 是否始终会话临时有效，或允许用户显式记住到项目
- 2 MB 单文件、20 MB 目录和 200 文件上限是否合适
- PDF、Word 和 Excel 解析分别放在哪个阶段
- 工具调用日志保留哪些参数才能兼顾诊断和隐私
- 默认交付模式是否始终为 `chat`，或允许按会话记住上次选择
- 图片、视频、音乐和语音各自首批支持哪些模型与 Provider
- 媒体任务默认并发 2 个是否需要按 Provider 单独配置
- 费用预估使用静态价格表、Provider 返回值还是只展示未知价格提示
- 聊天专属 Artifact 的回收区保留周期和空间配额是多少
- 声音克隆是否单独启用，并如何记录声音授权和使用范围
- 会话删除、项目删除和节点删除时的 Artifact 引用清理顺序
- `@model` 候选列表是否只显示与当前媒体工具兼容的模型

## 21. 下一步

下一步先确认 P0-A 的助手模型、流事件协议、会话 IndexedDB schema、Store revision 和批量删除事务。P0-A 稳定后，再分别确认 P1-A 的搜索 Provider 与安全网页读取器，以及 P1-B 的 FileGrant、格式白名单和大小限制。进入 P2 前，先确认默认交付模式、首批媒体 Provider、Artifact 生命周期和费用确认策略，再按“对话内生成 → 节点物化 → 双端交付”的顺序实施。每个阶段都要单独列出文件、测试、权限影响和回滚方式，不能一次性跨阶段开发。
